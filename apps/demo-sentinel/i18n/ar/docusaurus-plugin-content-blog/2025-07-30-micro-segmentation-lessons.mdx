Rampart هو طبقة التجزئة الدقيقة في Sentinel. يُقسّم الشبكة إلى مناطق، ويُشفّر حركة البيانات بينها، ويفرض سياسات الوصول عند كلّ حدّ منطقة. شغّلناه في الإنتاج لأكثر من عام. وفي ذلك الوقت، ارتكبنا ثلاثة أخطاء جوهريّة، كلّ واحد منها درس في كيفيّة خطأ التجزئة الدقيقة حين تلتقي النظريّة بشبكة فعليّة.

## الخطأ الأوّل: مناطق أكثر من اللازم

أنشأ نشر Rampart الأوّلي منطقة واحدة لكلّ خدمة دقيقة. بدا المنطق سليماً، أقصى عزل يعني أقصى أمان. كلّ حدّ خدمة حدّ تجزئة. ولا تستطيع أيّ خدمة الوصول إلى أيّ خدمة أخرى دون سياسة عبور صريحة بين المنطقتَين.

عملياً، أنشأنا 87 منطقة لـ 87 خدمة دقيقة. كان لرسم خرائط السياسة 340 قاعدة عبور. طلب واحد من الواجهة الأماميّة إلى صفحة تأكيد الدفع عبر 6 حدود مناطق. زاد زمن الانتقال 18 ميلي ثانية لكلّ طلب، ليس من عبء التشفير، بل من 6 تقييمات سياسة متتابعة عند كلّ حدّ.

دمجنا إلى 14 منطقة منظّمة بحسب مستوى الثقة وحساسيّة البيانات:

```text title="Zone consolidation"
Before: 87 zones, 340 crossing rules, 6 hops per request
After:  14 zones, 48 crossing rules, 2 hops per request

Zone structure:
  edge           → load balancers, CDN origin
  web            → frontend services
  api            → backend API services
  data-layer     → databases, caches
  analytics      → ETL, reporting
  payments       → PCI-scoped services
  auth           → identity, SSO integration
  admin-tools    → internal dashboards
  ci-cd          → build and deploy
  monitoring     → Watchtower, Spyglass
  control-plane  → Sentinel management
  staging        → pre-production mirror
  sandbox        → development, testing
  external       → third-party integrations
```

القاعدة العامّة التي وصلنا إليها، ينبغي أن تتبع حدود المناطق حدود الثقة، لا حدود الخدمات. الخدمتان اللتان تتعاملان مع الفئة ذاتها من البيانات وتتشاركان سياسات الثقة ذاتها تنتميان إلى المنطقة ذاتها.

## الخطأ الثاني: قواعد عبور بين المناطق متماثلة

في وقت مبكّر، كتبنا قواعد عبور بين المناطق ثنائيّة الاتجاه افتراضياً. إن استطاعت `api` الوصول إلى `data-layer`، فإنّ `data-layer` تستطيع الوصول إلى `api`. بدا هذا معقولاً، الخدمات تحتاج إلى التواصل في الاتجاهَين.

كان طريقاً سريعاً للتنقّل الجانبي. تستطيع قاعدة بيانات مُخترَقَة أن تبدأ اتصالات راجعة إلى طبقة API، التي تستطيع الوصول إلى طبقة الويب، التي تستطيع الوصول إلى الحافة. كان المسار بأكمله مُصرَّحاً لأنّ كلّ قاعدة عبور بين منطقتَين كانت متماثلة.

انتقلنا إلى قواعد أحاديّة الاتجاه:

```text title="trust-policy.grain — unidirectional zone crossing"
policy "api-to-data" {
  resource   = "data-layer"
  effect     = "allow"
  direction  = "outbound"

  conditions {
    zone.origin = "api"
    zone.target = "data-layer"
  }
}

// No corresponding data-layer-to-api rule exists.
// Response traffic flows back through the established connection.
// New connections from data-layer to api are denied.
```

تسري حركة بيانات الردّ على الاتصالات القائمة. تُحجب الاتصالات الجديدة في الاتجاه العكسي. يُلغي هذا أكثر أنماط التنقّل الجانبي شيوعاً، خدمة مُخترَقَة في منطقة أقلّ ثقة تبدأ اتصالات إلى منطقة أعلى ثقة.

## الخطأ الثالث: تجاهل DNS

جزّأنا الشبكة لكنّنا تركنا حلّ DNS غير مُجزَّأ. استطاعت كلّ منطقة حلّ كلّ اسم مضيف. ولم تستطع خدمة مُخترَقَة في منطقة `sandbox` الاتصال بموارد `control-plane`، لكنّها استطاعت حلّ أسماء مضيفيها ومعرفة عناوين IP الخاصّة بها.

سرّب هذا الطوبولوجيا الداخليّة لأيّ مهاجم يحصل على موطئ قدم. كان بإمكانهم رسم الشبكة بأكملها دون إرسال اتصال واحد يعبر حدّ منطقة.

نشرنا عرض DNS لكلّ منطقة. تحلّ كلّ منطقة فقط أسماء مضيفي الموارد المُصرَّح لها بالوصول إليها:

- `sandbox` تحلّ موارد sandbox فقط.
- `api` تحلّ موارد data-layer وauth وexternal.
- `control-plane` تحلّ كلّ شيء.

DNS الآن حدّ تجزئة، لا مجرّد خدمة تسمية. إذا لم تستطع منطقة حلّ اسم مضيف، فلا تستطيع استهداف ذلك المورد، ولا حتى للاستطلاع.

## الخلاصة

التجزئة الدقيقة ليست تمريناً طوبولوجياً. إنّها تمرين ثقة. ينبغي للمناطق أن تعكس حدود الثقة، وأن تكون القواعد أحاديّة الاتجاه، وأن تحترم البنية التحتيّة الداعمة، DNS واكتشاف الخدمات وسلطات الشهادات، الحدود ذاتها. وإلّا، فالتجزئة خريطة بها ثقوب، والمهاجمون بارعون جدّاً في إيجاد الثقوب.

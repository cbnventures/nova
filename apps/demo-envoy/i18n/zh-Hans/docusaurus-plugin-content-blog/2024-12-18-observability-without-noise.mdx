我们曾拥有 14000 个指标、12 个仪表盘、4 条告警通道。某个星期四的 23:23，Spoke 端点宕机，竟然没人在四十分钟内察觉——信号被噪声淹没了。

问题从来不是可观测性不足，而是过度。

## 噪声从何而来

Courier 中的每一条中继都会发出一堆指标：接收时间、认证耗时、转换耗时、路由耗时、投递耗时、载荷大小、响应码、重试次数、目的地、来源、事件类型……把这些乘以每秒 30000 次中继，你就得到了一条数据消防水龙——技术上无所不包，实际上毫无用处。

我们的 Glassboard 仪表盘看上去很气派，几乎为一切都画了曲线。但"一切"也意味着，当某个目的地的投递延迟尖峰时，它只是一张被三天没人看过的图表上四十条曲线中的一条。

```text title="告警量 — 2024 年第三季度"
Month     Alerts Fired    Actionable    False Positives    Missed Incidents
──────────────────────────────────────────────────────────────────────────
July           312             41              267                4
August         287             38              244                3
September      341             52              282                7
──────────────────────────────────────────────────────────────────────────
Total          940            131              793               14
```

940 条告警里只有 131 条是可执行的——14% 的信号率。其余 86% 训练值班工程师学会了忽略通知。

## 三条规则

我们从零开始重写了告警体系，只立了三条规则：

1. **报症状，不报原因。** 别在 CPU 高时报警，要在投递延迟违反 SLA 时报警。CPU 是原因，延迟是症状，客户感受到的是症状。
2. **一条告警，一条动作。** 每条告警都必须有清晰、文档化的应对方式。若应对方式是"打开仪表盘自己看着办"，那这条告警还没准备好。
3. **沉默是一种信号。** 90 天没有触发过的告警，删掉它。从不触发的告警，要么是配错了，要么在监控不会坏的东西——无论哪种，都是噪声。

## 新的栈

Glassboard 现在为每个目的地只追踪五个聚合指标：

- **投递速率**（每秒消息数）
- **错误率**（每秒非 2xx 响应数）
- **延迟**（投递 p50 与 p99）
- **队列深度**（等待投递的消息数）
- **死信计数**（过去一小时内耗尽重试的消息数）

就这些，每个目的地五个数字。其中任意一个越过阈值，Glassboard 就通过 Courier 触发一条告警，附带足够诊断问题的上下文——无需打开任何仪表盘。

```text title="Glassboard 告警载荷 — 通过 Courier 投递"
{
  "severity": "critical",
  "destination": "spoke://crm-sync",
  "trigger": "error_rate",
  "value": 42.3,
  "threshold": 5.0,
  "window": "60s",
  "context": {
    "delivery_rate": 180.7,
    "latency_p99": "340ms",
    "queue_depth": 1247,
    "dead_letters_1h": 89,
    "top_error": "503 Service Unavailable (78%)",
    "sample_message_id": "msg_f8a2c1d4"
  }
}
```

值班工程师只会收到一条通知，它告诉对方哪儿坏了、坏到什么程度、从哪儿开始查。分诊阶段不需要再去打开仪表盘。

## 结果

重写完成后，我们把新旧系统并行跑了 30 天：

| 指标         | 旧告警    | 新告警    |
|------------|--------|--------|
| 每月告警数      | 312    | 23     |
| 可执行率       | 14%    | 91%    |
| 漏检事故       | 4      | 0      |
| 平均检测时间     | 11 分钟  | 47 秒   |
| 值班疲劳（问卷调查） | 7.2/10 | 2.1/10 |

每月 23 条告警，其中 21 条可执行，零次漏检。平均检测时间从 11 分钟降到 47 秒。

## 教训

可观测性不是关于采集数据，而是关于在合适的时候把对的数据浮上来。每加一个指标、配一条告警、开一条通知通道，都在消耗注意力——而值班轮值中最稀缺的资源，就是注意力。

五个指标、一条告警、足够采取行动的上下文。这就是整套可观测性哲学。

绝大多数组织只在边界做加密。TLS 终止在负载均衡器上，背后的一切都以明文流动。其假设是内部流量是安全的——因为它从不离开数据中心。这个假设错了，而且错了很久。

东西向流量——服务之间在内网中的流量——正是横向移动发生的地方。攻击者一旦攻陷某个服务，就可以监听到邻近服务的明文流量、收割凭证、提权扩散，从未触及外围。加密这部分流量不是可选项，问题是：怎样做才不至于把吞吐打死。

## 性能问题

最朴素的做法，是在每一对服务之间启用双向 TLS。小规模下这能跑。大规模下，它会带来三个问题：

1. **握手开销。** 每条新连接都需要一次 TLS 握手。在每次请求要发起数百次服务间调用的微服务架构中，累计的握手延迟是可测量的。
2. **证书管理。** 每个服务都需要证书，证书会过期，轮换需要协调。一张过期证书就能引发雪崩。
3. **CPU 饱和。** TLS 的加解密消耗 CPU 周期。在每秒处理上万次请求的高吞吐服务上，按请求加密的 CPU 开销不容忽视。

在部署 Rampart 加密层之前，我们在一份生产负载上测量了影响：

| 指标         | 明文   | 朴素 mTLS | Rampart |
|------------|------|---------|---------|
| P50 延迟（毫秒） | 2.1  | 4.8     | 2.4     |
| P99 延迟（毫秒） | 12.3 | 31.7    | 14.1    |
| CPU 利用率    | 34%  | 61%     | 38%     |
| 握手速率（次/秒）  | 0    | 8,400   | 340     |
| 证书数量       | 0    | 214     | 12（按区域） |

差距源于架构。朴素 mTLS 为每条连接建立一次新的 TLS 会话；Rampart 在区域边界之间建立长生命周期的加密隧道，并在隧道上复用服务流量。

## Rampart 如何处理

Rampart 的加密模型工作在区域级别，而非服务级别。它不为每对服务间连接单独加密，而是在区域网关之间建立持久的加密隧道。所有跨越区域边界的流量都流经这些隧道。

```text title="Rampart 区域隧道拓扑"
┌──────────────┐    加密隧道            ┌──────────────┐
│  payments    │◄──────────────────────►│  data-layer  │
│  区域         │                        │  区域         │
│              │    加密隧道            │              │
│  svc-pay ──►─┤◄──────────────────────►├─► svc-cache  │
│  svc-bill ─►─┤                        ├─► svc-db     │
└──────────────┘                        └──────────────┘
```

隧道底层使用 WireGuard。握手发生在隧道建立时，而不是每条连接上。证书管理也简化为每个区域一个身份，而不是每个服务一个。共享同一隧道的成千上万条连接摊薄了 CPU 开销，因此整体下降明显。

## 我们加密什么

全部。Rampart 不区分敏感与非敏感的东西向流量。加密层对应用透明——服务彼此之间仍然使用过去那一套地址与端口。区域网关拦截流量，加密它，然后通过隧道转发。

这从根本上消除了分类问题。你不必决定哪些流量值得加密；你把所有流量都加密了，而性能代价微乎其微——因为隧道架构摊薄了开销。

## 取舍

区域级加密不提供同一区域内的服务级隔离。同一个 Rampart 区域内的两个服务之间仍以明文通信。这是有意为之——区域内流量由信任策略管控，而非加密。如果你需要同一区域内的两个服务之间也加密，那就该把它们拆到不同区域中。

经验法则是：如果两个服务不应能读到对方的流量，它们就该在不同的 Rampart 区域中。区域边界既是分段边界，也是加密边界。两者是同一回事。

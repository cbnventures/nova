On September 29th at 16:42 UTC, a single misconfigured Parcel file caused a cascade failure that affected four customers, dropped 12,000 messages over 23 minutes, and triggered the longest incident in Envoy's history.

This is the full post-mortem. No details redacted.

Total duration: 23 minutes 30 seconds. Messages dropped: 12,147. Customers affected: 4 (all sharing envoy-prod-01 and envoy-prod-02 during the traffic redistribution).

## Root Cause

Customer D's `.parcel` file contained a circular field reference:

```text title="The bad .parcel file (simplified)"
provider "internal-webhook" {
  extract {
    user_id   = "$.data.user.id"
    user_data = "$.data.user"
    nested    = "$ref:user_data.nested"   # ← references itself via user_data
  }
}
```

The `$ref:user_data.nested` extraction referenced `user_data`, which contained the full `$.data.user` object, which contained a `nested` field that pointed back to the same path. The Parcel parser followed the reference chain without a depth limit and entered an infinite loop.

## What Went Wrong

Three things failed:

1. **No validation of circular references.** The Parcel parser validated syntax but not reference topology. A circular reference was syntactically valid.
2. **No CPU isolation between customers.** All customer configs ran on shared Courier instances. One customer's bad config consumed 100% CPU on the shared instance.
3. **Restart loop.** Trellis restarted the failed instance, which loaded the same bad config, which triggered the same infinite loop. The restart was not a recovery — it was a repetition.

## What We Fixed

### Fix 1: Reference Depth Limit

Parcel now enforces a maximum reference depth of 8. Any `$ref` chain deeper than 8 levels is rejected at config load time with a clear error message.

### Fix 2: Config Validation Gate

New relay configs are now validated in a sandboxed Vial instance before being applied to production. The sandbox processes a synthetic webhook through the full pipeline. If it crashes, loops, or exceeds a 5-second timeout, the config is rejected.

### Fix 3: Per-Customer CPU Budgets

Courier now tracks CPU time per customer config. If a single customer's relays consume more than 40% of available CPU for more than 10 seconds, Courier automatically suspends that customer's config and fires a P1 alert with the customer ID and config hash.

### Fix 4: Config-Aware Restarts

Trellis now checks whether a failed instance was processing the same config that caused the previous failure. If the config has not changed, Trellis does not restart — it cordons the instance and alerts the operator instead.

All dropped messages were recovered from Ledger's write-ahead log and replayed within two hours of the incident.

## The Lesson

Multi-tenant infrastructure fails in multi-tenant ways. A bug in one customer's config should not be able to affect another customer's traffic. Isolation is not optional — it is the boundary between "one customer has a bad day" and "four customers have a bad day."

We violated that boundary. We will not violate it again.

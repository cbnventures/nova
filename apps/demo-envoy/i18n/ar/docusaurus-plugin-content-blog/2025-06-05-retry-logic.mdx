أصعب ممّا تظنّ. أصعب بكثير.

## النهج الساذج

أبسط إعادة محاولة هي إعادة محاولة فورية. فشل الطلب، فترسله مرّة أخرى الآن. هذا يعمل مرّة واحدة بالضبط — حين كان الفشل خللًا شبكيًّا عابرًا. في أيّ سيناريو آخر، يجعل الأمور أسوأ.

```text title="Immediate retry under load"
03:47:00  POST /relay — 503 Service Unavailable
03:47:00  POST /relay — 503 Service Unavailable (retry 1)
03:47:00  POST /relay — 503 Service Unavailable (retry 2)
03:47:00  POST /relay — 503 Service Unavailable (retry 3)
# Destination is overloaded. You just made it worse.
```

الخطوة التالية إعادة محاولة بفاصل ثابت. انتظر ثانية بين المحاولات. أفضل، لكنّها لا تزال إشكالية. إن كانت مئة عملية ترحيل تُعيد المحاولة على إيقاع الثانية الواحدة نفسه، تحصل على قطيع رعدي متزامن يُبقي الوجهة مُثبّتة.

## ماذا يفعل Courier فعلًا

يستخدم Courier تراجعًا أُسّيًّا مع رجفة. يزيد التراجع التأخير بين المحاولات. والرجفة تعشوي التوقيت الدقيق كي لا تتزامن إعادة المحاولات المتزامنة.

| الاستراتيجية | نمط التأخير                | خطر القطيع الرعدي | ضمان التسليم |
|--------------|----------------------------|-------------------|--------------|
| فورية        | 0s, 0s, 0s                 | حرج               | لا يوجد      |
| فاصل ثابت    | 1s, 1s, 1s                 | عالٍ              | ضعيف         |
| تراجع أُسّي  | 1s, 2s, 4s, 8s             | متوسّط            | معتدل        |
| تراجع + رجفة | 0.8s, 2.3s, 3.7s, 9.1s     | منخفض             | قويّ         |
| Courier      | تراجع + رجفة + رسائل ميّتة | ضئيل              | مضمون        |

السطر الأخير هو موضع Courier. التراجع والرجفة يتعاملان مع التوقيت. لكن التسليم المضمون يتطلّب قطعتين أخريين: طوابير الرسائل الميّتة وإيصالات التسليم.

## طوابير الرسائل الميّتة

الرسالة التي استنفدت كلّ محاولات إعادة الإرسال لا تُحذف. بل تُنقَل إلى طابور رسائل ميّتة — منطقة احتجاز للرسائل التي تعذّر تسليمها ضمن نافذة إعادة المحاولة.

```text title="Courier dead-letter flow"
Attempt 1  →  503  →  wait 1.2s
Attempt 2  →  503  →  wait 4.7s
Attempt 3  →  503  →  wait 28.3s
Attempt 4  →  503  →  wait 2m 10s
Attempt 5  →  503  →  DEAD LETTER

Message moved to: /ledger/dead-letter/msg_a7f3c9
Escalation: Dispatch routes to fallback destination
```

الرسائل الميّتة قابلة للاستعلام عبر Spoke API. يستطيع المُشغّلون فحصها، أو إعادة محاولتها يدويًّا، أو حذفها. يُسجّل Ledger كلّ تحوّل حالة.

## إيصالات التسليم

لا يعتبر Courier أن الرسالة سُلِّمت حين تُعيد الوجهة 200. بل يعتبر الرسالة مُسلَّمة حين تُعيد الوجهة 200 *و* يُطابق محتوى الاستجابة مخطّط الإيصال المتوقّع.

هذا التمييز مُهمّ. وكيل عكسي يُعيد 200 بمحتوى فارغ ليس مثل الخدمة الفعلية تؤكّد الاستلام. يُصادق Courier الإيصال، ويُسجّله في Ledger، وعندئذٍ فقط يضع علامة اكتمال التسليم.

### المخطّط الكامل لإعداد إعادة المحاولات في Courier

```text title="relay.grain — retry block"
retry {
  # Strategy: "exponential" | "fixed" | "immediate"
  strategy = "exponential"

  # Maximum retry attempts before dead-letter
  max = 5

  # Base delay for exponential backoff
  base_delay = "1s"

  # Maximum delay cap
  max_delay = "10m"

  # Jitter range (percentage of current delay)
  jitter = 0.25

  # Dead-letter queue configuration
  dead_letter {
    enabled = true
    retention = "30d"
    escalation = "dispatch://fallback-channel"
  }

  # Delivery receipt validation
  receipt {
    required = true
    schema   = "standard"
    timeout  = "5s"
  }
}
```

## الدرس

منطق إعادة المحاولة ليس حلقة باستدعاء sleep. بل آلة حالات بخمس حالات على الأقلّ: قيد الانتظار، يحاول، يتراجع، رسالة ميّتة، مُسلَّم. لكلّ انتقال قواعد. ولكلّ حالة آثار جانبية. ولكلّ نمط فشل استجابة صحيحة مختلفة.

يتعامل Courier مع هذا فلا يحتاج بيان الترحيل إلى ذلك. تُعلن سياسة إعادة المحاولة في ملفّ `.grain` الخاصّ بك. ويتولّى Courier البقيّة.

ذلك هو المغزى. الموثوقية لا ينبغي أن تتطلّب خبرة عند كلّ نقطة تكامل. بل ينبغي أن تكون الافتراض.

计算机科学有一句出名的玩笑：业内只有两件难事——缓存失效、命名以及 off-by-one。我们解决第一件的方式，是干脆拒绝去做"失效"。Quench 不让缓存条目失效，它只创建新条目。旧条目会过期。整个缓存只追加、不可变。

这不是什么新点子。内容寻址存储自 Git 起就存在。但要把它套到构建缓存上——这里的"内容"是源码、依赖版本、编译参数与平台目标的组合——就得重新思考"缓存键"到底意味着什么。

## 缓存键

Quench 的缓存键，是按确定顺序拼接的四个输入的 SHA-256 哈希：

1. **源码哈希。** 包内每一个文件按路径排序后单独哈希，再用 Merkle 树合并成一个根哈希。
2. **依赖哈希。** Tongs 图中每条依赖（含传递依赖）解析后的版本。
3. **清单哈希。** `.grain` 清单的相关片段——Warden 规则、Smelter 目标、平台开关。
4. **工具链哈希。** Foundry 自身的版本，以及流水线中每个编译器和工具的版本。

任何一项发生变化，缓存键就变化，Quench 会把它视为一次全新构建。如果都不变，缓存产物就会被逐字节恢复。

```text title="Quench 缓存键分解"
$ foundry quench key --verbose packages/core

源码哈希：    a1b2c3d4e5f6...
依赖哈希：    f6e5d4c3b2a1...
清单哈希：    1a2b3c4d5e6f...
工具链哈希：  6f5e4d3c2b1a...
─────────────────────────────────
缓存键：      9x8y7z6w5v4u...
```

## 为什么不用时间戳？

大多数构建工具用文件修改时间戳来判断是否需要重建。这样做很快，但是错的。时间戳至少在三种常见情境下会失灵：

- **Git 操作。** 检出一个分支会改掉所有差异文件的 mtime，即使你切回去也一样。明明没有真正变化的文件，构建缓存却已经过期。
- **CI 环境。** 持续集成的 runner 通常会重新克隆仓库，于是所有 mtime 都被设为克隆时间。从构建工具的角度看，每一个文件都刚刚变更过。
- **时区与时钟漂移。** 分布式团队若时钟不同步，可能产生未来时间戳，超过缓存产物的时间。构建工具看到"较新"的源文件，便不必要地重建。

改用对文件内容求哈希，三种问题一并消失。内容相同的两个文件产生相同的哈希，与谁修改、何时修改、机器在什么时区毫无关系。

## 远程缓存共享

默认情况下，Quench 把缓存产物存在本地目录。但因为缓存键是确定性的，跨机器共享非常自然。如果开发者 A 构建了包 `core` 并把缓存产物推送到共享存储，开发者 B 只要输入产生相同的缓存键，就可以直接拉取并跳过构建。

我们支持三种远程后端：

- **文件系统（NFS/SMB）。** 挂载一个共享目录，Quench 像对待本地缓存一样读写。
- **对象存储。** S3 兼容的端点。每个缓存条目都是按其哈希为键的单个对象。
- **Foundry Cloud。** 我们为自己团队运行的托管服务，负责存储、淘汰与访问控制。

远程缓存把我们的 CI 构建时间又减少了 40%，因为大多数 pull request 构建的缓存键有 95% 与主分支重合。

## 淘汰

Quench 不会在构建期间删除缓存条目。淘汰是另一个独立的离线过程。默认策略是 LRU，TTL 7 天，容量上限 10 GB。两者都可以在清单里调整：

```text title="project.grain — 缓存淘汰策略"
workspace "my-workspace" {
  quench {
    max_size = "20GB"
    max_age  = "14d"
  }
}
```

淘汰会在一次成功构建结束后自动运行，也可以通过 `foundry quench prune` 手动触发。它从不在构建中途执行——因为在构建过程中移除一条缓存条目，可能导致下游包错过它本该命中的缓存。

## 教训

当你试图弄清楚"什么变了"，缓存失效是难题；当你给一切都打哈希、让哈希自己说话，缓存失效就只是琐事。

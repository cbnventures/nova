Bellows 是 Foundry 的并行任务运行器。它读取 Tongs 的依赖图，判断哪些包可以同时跑，然后把工作分摊到每一个可用的 CPU 核心。多数团队切到 Bellows 之后能看到 3x 到 5x 的构建时间提升。不过，调度算法远比"独立包并行跑"复杂。

并行任务调度有三类难题：内存压力、任务异质性、依赖链深度。Bellows 都处理了。理解它的工作方式，能帮助你为最大化吞吐去规划工作空间。

## 约束 1：内存压力

朴素的做法是尽量多并行——有 8 核就同时跑 8 个包。一开始可行，直到其中某个包是要吃 2 GB 内存的 Smelter 编译。突然之间，8 个编译开始抢内存，操作系统开始换页，构建反而比串行更慢。

Bellows 给每种任务类型分配了"内存权重"。Smelter 编译是重量级，Warden 检查是轻量级，Crucible 测试介于两者之间。调度器跟踪估算的内存总量，一旦新任务会把总量推过可用 RAM 的 80%，就不会启动它。

```text title="Bellows 调度器输出"
$ foundry ignite --verbose

[bellows] 可用核心：8，内存：32GB，上限：25.6GB
[bellows] 第 1 波：core (build, 1.8GB), shared-types (build, 0.4GB),
          utils (build, 0.6GB), constants (build, 0.2GB)
          → 4 个任务，预估 3.0GB
[bellows] 第 2 波：api (build, 2.1GB), web (build, 2.4GB),
          core (test, 0.8GB), core (lint, 0.3GB)
          → 4 个任务，预估 5.6GB
[bellows] 第 3 波：gateway (build, 1.9GB), dashboard (build, 1.6GB),
          api (test, 0.9GB), web (lint, 0.3GB)
          → 4 个任务，预估 4.7GB
```

注意第 2 波在 `api` 和 `web` 构建的同时，已经开始跑 `core` 的测试与检查了。Bellows 不会等所有构建结束再开始测试——它会在前置条件一就位时立刻把不同类型的任务穿插起来。

## 约束 2：任务异质性

不同任务的耗时差别极大。一次 Smelter 编译，大包可能要 12 秒，小到几乎纯常量的包可能只要 0.3 秒。如果天真地排程，关键路径会被大包统治，而小包瞬间完成、核心闲置。

Bellows 采用 longest-processing-time-first（LPT）的启发式策略：根据 Quench 中的历史数据估算每个任务的时长，然后优先调度最长的任务。这样可以让核心在整个构建过程中持续忙碌，并把尾部延迟——也就是倒数第一与倒数第二之间的间隔——降到最小。

## 约束 3：依赖链深度

在一个深依赖链 monorepo 里——A 依赖 B、B 依赖 C、C 依赖 D——最大并行度受限于链的长度，而非核心数。如果链有 7 级深，无论核心多少，总会至少存在 7 个串行步骤。

Bellows 无法打破这条限制，但它能缓解。当存在深链时，Bellows 会优先调度它——关键路径上的包总是最先排上，较短的链填补在它周围。这就是项目管理工具里"关键路径调度"用的思路。

## 你能做什么

工作空间的布局直接影响 Bellows 能榨出多少并行度。三种结构性调整能稳定地提升构建时间：

- **压平依赖图。** 如果包 A 只是因为某个类型而依赖 B，可以考虑把那个类型搬到共享包里，让 A 与 B 都依赖它。这能从链上摘掉一节，让 A 与 B 并行构建。
- **拆分大包。** 一个编译要 30 秒的包就是瓶颈。如果里头的模块互相独立，把它拆成两个包就能让 Bellows 并发编译。
- **分离测试依赖。** 如果你的测试夹具放在生产代码同一包里，Bellows 就只能等生产构建结束才能启动测试。把夹具搬到一个独立包里，它就能与主包并行构建。

这些都不是 Bellows 专属优化，而是良好的 monorepo 架构。Bellows 只是把回报变得可见，因为你能在构建输出中直接看到调度波次。

## 衡量并行度

运行 `foundry ignite --profile` 可以生成一份构建画像：每个任务的开始时间、时长、跑在哪个核心。把它导入任意 trace viewer，就能直观看到调度波次。

我们跟踪的指标是 **parallel efficiency**：总 CPU 时间与挂钟时间之比。一台完美并行的 8 核机器，理论上能达到 8.0x。在我们 200 包的工作空间，实际能跑到 5.8x；剩下的 2.2x 损耗在依赖链与内存调度上。

这个数字已经足够好了：在 8 核机器上拿到 5.8x，意味着 Bellows 用掉了 72% 的可用算力。剩下的部分都在等依赖图，任何调度器都消除不了。

كنا نبني مساحات عمل متعدّدة الحزم منذ نحو عقد. ولِنحو عقد، كنا نغرق في ملفات إعداد لا يصونها أحد.

{/* truncate */}

## الانحراف الذي لا يلاحظه أحد

كل مساحة عمل تبدأ نظيفة. حزمة واحدة، وإعداد Warden واحد، وملف إعدادات Alloy واحد. ثم تظهر الحزمة الثانية، وينسخ أحدهم الإعداد من الأولى. بحلول الشهر السادس، لديك اثنتا عشرة حزمة، وثلاث إعدادات Warden مختلفة، وإعدادان متعارضان لـ Alloy، وBlueprint نسخه أحدهم ولصقه من حزمة core ولم يحدّثها قط.

نُسمّي هذا انحراف الإعدادات. هو التباعد البطيء والصامت لإعدادات البناء عبر قاعدة الشيفرة. لا يلاحظه أحد حتى يسأل موظف جديد لماذا تسمح حزمة `api` بـ `any` ضمني بينما لا تسمح حزمة `web`.

## كيف تبدو اثنا عشر ملف إعداد

هكذا كانت تبدو مساحة عمل نموذجية قبل Foundry. كان لكل حزمة مجموعتها الخاصة من ملفات الإعداد:

```text title="Before: 12 config files per workspace"
my-workspace/
  .wardenrc.grain
  alloy.config.grain
  smelter.config.grain
  blueprint.grain
  packages/
    core/
      .wardenrc.grain
      alloy.config.grain
      smelter.config.grain
    api/
      .wardenrc.grain
      alloy.config.grain
      smelter.config.grain
    web/
      .wardenrc.grain
      alloy.config.grain
      smelter.config.grain
```

اثنا عشر ملفًا. أربع حزم. وكل واحد منها يمكن أن ينحرف بشكل مستقل.

## كيف يبدو بيان واحد

هذه هي مساحة العمل ذاتها موصوفة في بيان `.grain` واحد:

```text title="After: 1 manifest"
# project.grain
workspace "my-workspace" {
  lang    = "alloy"
  warden  = ["strict", "conventions"]
  smelter = "standard"

  packages {
    core { type = "library" }
    api  { type = "service", depends = ["core"] }
    web  { type = "app", depends = ["core"] }
  }
}
```

ملف واحد. مساحة العمل ذاتها. كل حزمة ترث قواعد Warden ذاتها، وإعدادات Alloy ذاتها، وإعداد Smelter ذاته. إن احتجت إلى تجاوز شيء لحزمة بعينها، تفعله داخل السطر:

```diff title="project.grain — per-package override"
  packages {
    core { type = "library" }
-   api  { type = "service", depends = ["core"] }
+   api  { type = "service", depends = ["core"], warden = ["strict", "conventions", "api-safety"] }
    web  { type = "app", depends = ["core"] }
  }
```

التجاوز صريح. يعيش بجوار الشيء الذي يُعدِّله. لا يوجد ملف مخفي في دليل متداخل يُغيِّر السلوك بصمت.

:::tip
شغِّل `foundry ignite --dry-run` لرؤية ما سيُولِّده Foundry دون كتابة أي ملفات. هذه أسرع وسيلة للتحقق من بيانك قبل الإيداع.
:::

## الأطروحة

ينبغي أن تكون النيّة تصريحية. ينبغي أن تصف كيف تبدو مساحة عملك — اللغات، والتبعيات، والقواعد، والأهداف — وأن تكتشف سلسلة الأدوات الباقي.

ملفات الإعداد تفاصيل تنفيذ. هي ناتج جانبي لسلسلة أدوات لا تستطيع استنتاج ما تعنيه من ما كتبته. كلّما عدّلت إعداد Warden يدويًا، فأنت تقوم بعمل كان يمكن للمُترجم القيام به نيابة عنك.

لهذا بَنينا Foundry. ليس لأننا أردنا CLI آخر. بل لأننا أردنا التوقّف عن كتابة الملفات الاثني عشر ذاتها في كل مشروع والبدء بوصف ما نعنيه فعلًا.

## ما تغيّر بالنسبة لنا

منذ تحويل مساحات عملنا الداخلية إلى بيانات Foundry:

- **انخفضت العلل المتعلقة بالإعداد إلى صفر.** لا مزيد من «يعمل على جهازي» بسبب إعداد Alloy قديم في حزمة متداخلة.
- **انخفض زمن الإعداد من ساعتين إلى خمس عشرة دقيقة.** يقرأ المهندسون الجدد بيانًا واحدًا بدلًا من البحث عبر اثني عشر ملف إعداد.
- **انتقل اتّساق قواعد Warden من 74% إلى 100%.** كل حزمة تُشغِّل القواعد ذاتها. لا استثناءات إلا إذا قال البيان ذلك.

المسبك لا يهتم بآرائك. يهتم ببيانك. اكتبه مرّة، وكل ما يأتي بعده يبقى متوافقًا.

## جرّبه

```bash
spark install foundry
foundry ignite
```

أمران. بيان واحد. صفر ملفات إعداد. اقرأ دليل [نظرة عامة](/docs/overview/) لترى كيف يعمل من الصفر.

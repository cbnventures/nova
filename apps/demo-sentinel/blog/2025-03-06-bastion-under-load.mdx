Bastion is the access gateway layer in Sentinel. Every connection from an enrolled device to a protected resource flows through a Bastion node. The node enforces the trust decision from Citadel, terminates the encrypted tunnel from the device agent, and proxies the connection to the target resource. It is the chokepoint by design. That means it has to survive being a chokepoint.

Last quarter we ran a capacity test to find out what happens when a single Bastion node hits 50,000 concurrent connections. The answer was more nuanced than we expected.

{/* truncate */}

## The Test Setup

We provisioned a single Bastion node on a `c6g.4xlarge` instance (16 vCPU, 32 GB RAM) and aimed a load generator at it. The generator simulated enrolled devices establishing WireGuard tunnels and sending HTTP requests through the gateway to a backend target pool.

The test ramped connections linearly from 0 to 60,000 over 30 minutes, held steady for 15 minutes, then drained.

## The Results

```text title="Bastion capacity test — single node"
Phase 1: Ramp (0 → 60,000 connections over 30 minutes)

 Connections   P50 (ms)   P99 (ms)   CPU    Memory   Drops
      5,000       1.2        4.1     11%     3.2 GB     0
     10,000       1.3        4.4     18%     5.8 GB     0
     20,000       1.4        5.2     31%     10.1 GB    0
     30,000       1.6        7.8     44%     14.7 GB    0
     40,000       2.1       12.3     58%     19.2 GB    0
     50,000       3.4       22.7     73%     23.8 GB    0
     55,000       8.1       89.4     86%     27.1 GB    14
     60,000      14.7      204.1     97%     30.4 GB   347

Phase 2: Steady state at 60,000 (15 minutes)
  Mean P50: 16.2 ms | Mean P99: 241.3 ms | Drops: 1,204

Phase 3: Drain
  Clean shutdown, all connections terminated gracefully.
```

The inflection point is clear. Below 50,000 connections, latency stays well within acceptable bounds. Between 50,000 and 55,000, P99 latency begins to spike. Above 55,000, the node is saturated and starts dropping connections.

## Where the CPU Goes

We profiled the node during the test. The CPU breakdown at 50,000 connections:

- **WireGuard encryption/decryption**: 38%
- **Policy evaluation cache lookups**: 19%
- **Connection proxy (TCP splice)**: 14%
- **TLS termination (backend)**: 12%
- **Logging and telemetry**: 9%
- **Kernel network stack**: 8%

Encryption dominates, which is expected. Every byte of traffic through a Bastion node is encrypted twice — once in the WireGuard tunnel from the device, once in the TLS connection to the backend. The double-encryption tax is the cost of zero-trust at the transport layer.

## The Scaling Model

Based on these numbers, our production recommendation is to provision Bastion nodes at a maximum of 40,000 concurrent connections per node. That leaves a 25% headroom buffer for traffic spikes and gives us a clear P99 target under 15 milliseconds.

For a fleet of 10,000 enrolled devices with an average of 8 concurrent connections per device, the math is:

```text title="Capacity planning"
10,000 devices × 8 connections = 80,000 total connections
80,000 / 40,000 per node       = 2 Bastion nodes (minimum)
                                 3 Bastion nodes (with N+1 redundancy)
```

We deploy Bastion nodes in regional clusters with health-check-based failover. If a node fails, its connections are redistributed to the remaining nodes in the cluster. The N+1 model ensures that a single node failure does not push the surviving nodes past the 40,000 threshold.

## What We Are Improving

The encryption overhead is the largest single cost. We are evaluating kernel-bypass networking (DPDK) and hardware-accelerated encryption (AWS Nitro Enclaves) to reduce the per-byte encryption cost. Early benchmarks suggest a 30-40% reduction in CPU utilization at the same connection count, which would push the comfortable ceiling from 40,000 to approximately 55,000 connections per node.

Until then, the scaling model is straightforward: more connections means more nodes. Bastion is designed to scale horizontally, and the node cost is predictable enough to capacity-plan with a spreadsheet.

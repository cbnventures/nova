我们只有 72 小时，要把一支企业级设备群从基于证书的 VPN 认证迁移到 Sentinel 的设备注册。一万台端点，横跨三大洲，零停机容忍。这是我们的复盘。

项目原本计划用六周完成。后来该客户的 VPN 集中器厂商宣布了一个比预期早两个月的产品退役日，时间表瞬间被压缩到一个周末。

## 注册流水线

Sentinel 的设备注册不是"装上代理就走"那种推送。每一台设备都必须注册硬件身份、完成态势证明，并获得一次 Bastion 网关绑定，之后才能参与任何信任评估。流程大致是这样：

```text title="注册流程"
1. 安装代理            → 通过 MDM 部署 sentinel-agent
2. 硬件证明            → 生成 TPM / Secure Enclave 密钥
3. 态势基线            → 记录操作系统版本、磁盘加密、防火墙状态
4. Bastion 绑定        → 将设备分配到最近的 Bastion 网关
5. 策略下发            → 设备从其设备组继承信任策略
```

每一步都必须成功，下一步才能开始。态势基线失败的设备拿不到 Bastion 绑定；没有 Bastion 绑定的设备就触不到任何受保护的资源。没有兜底路径。

## 哪里出了问题

迁移周末出了三件事：

- **1,247 台设备的态势基线失败。** 多数是操作系统版本落后最低要求一个小版本。我们把态势阈值设在了 `>=14.2`，而设备群里有一条长尾停留在 `14.1`。我们将阈值降到 `14.1` 并给了 30 天宽限期，同时把这些设备标记为强制升级。
- **APAC 区域的 Bastion 网关饱和。** 我们为悉尼办公室置备了 3 个 Bastion 节点，但注册风暴同时打到了三台节点上，连接排队超过了 30 秒的超时阈值。我们临时加开两个节点，重新跑了 APAC 的批次。
- **一处组策略冲突。** `contractors` 组里的设备同时继承了 `restricted-access` 与 `internal-services-baseline` 两条策略。两者交集为空——没有资源同时满足两条策略的条件——所以承包商设备虽然注册成功，但什么都触不到。我们为它们建了一条专门的 `contractor-services` 策略，显式地授予资源访问。

## 数字

| 指标          | 目标     | 实际       |
|-------------|--------|----------|
| 已注册设备数      | 10,000 | 9,814    |
| 注册失败数       | < 500  | 186      |
| 平均注册耗时      | < 5 分钟 | 3 分 42 秒 |
| Bastion 可用率 | 99.9%  | 99.7%    |
| 注册后支持电话数    | < 200  | 91       |

186 个失败全是硬件证明问题——TPM 或 Secure Enclave 被禁用的设备。这些需要 IT 现场介入，在第二周陆续完成。

## 我们会改什么

下一次，我们会把 Bastion 网关容量按预期并发注册速率的 3 倍预置，而不是 1.5 倍。注册风暴比稳态流量更尖锐，网关排队就是瓶颈。我们也会在迁移前一周做一次态势预扫，提前抓出落后的操作系统版本。

设备注册是零信任的地基。Sentinel 的每一次信任决策，都依赖于识别设备、了解其健康状态、知道它在哪。一个周末让一万台设备穿过那道闸门不算优雅，但成功了。从此往后，这一万台设备的每一秒，都会被每 90 秒持续地评估一次。

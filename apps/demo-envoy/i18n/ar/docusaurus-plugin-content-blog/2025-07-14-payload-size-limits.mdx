حاول عميل ترحيل حمولة خطّاف ويب بحجم 47MB عبر Courier يوم الثلاثاء الماضي. كان نظام CI/CD يُرسل سجلّ بناء كاملًا بصورة كتلة JSON في جسم خطّاف الويب. فشل الترحيل. وفتح العميل تذكرة دعم يطلب فيها رفع الحدّ.

## حدّ 256KB

يفرض Courier حدًّا أقصى لحجم الحمولة قدره 256KB لكلّ خطّافات الويب الواردة. أيّ حمولة أكبر من 256KB تُرفَض في مرحلة الاستقبال باستجابة `413 Payload Too Large`. ولا تدخل الرسالة خطّ المعالجة. لا مصادقة، لا تحويل، لا توجيه، لا محاولة تسليم.

ليس هذا قيدًا تقنيًّا. يستطيع Courier التعامل مع حمولات أكبر. الحدّ قرار تصميمي بثلاث مبرّرات.

### 1. قابلية التنبّؤ بالذاكرة

يعالج Courier الترحيلات في الذاكرة. تُحفَظ كلّ حمولة في مخزن مؤقّت من الاستقبال حتى التسليم. عند 30,000 ترحيل في الثانية، يعني حدّ 256KB أن Courier يحتاج إلى 7.3GB كحدّ أقصى من ذاكرة المخزن المؤقّت عند الذروة (بافتراض أن كلّ ترحيل بأقصى حجم، وهو ما لا يحدث أبدًا — متوسّط الحمولة 4.2KB).

```text title="Memory ceiling at various payload limits"
Limit       Peak Memory (30k RPS)    Avg Memory (4.2KB actual)
──────────────────────────────────────────────────────────────
256KB              7.3GB                     126MB
1MB               28.6GB                     126MB
10MB             286.0GB                     126MB
Unlimited             ?                      126MB (until it isn't)
```

سطر «بلا حدود» هو المشكلة. متوسّط حجم الحمولة لا يحميك. عميل واحد يُرسل حمولات 47MB عند 100 RPS سيستهلك 4.7GB وحده — يكفي لتجويع كلّ ترحيل آخر في خطّ المعالجة.

### 2. تكلفة إعادة المحاولة

كلّ إعادة محاولة تُعيد إرسال الحمولة كاملة. حمولة 4KB تُعاد محاولتها خمس مرّات تُكلّف 20KB من النطاق. حمولة 47MB تُعاد محاولتها خمس مرّات تُكلّف 235MB. اضرب ذلك في إعادة محاولات متزامنة عبر الوجهات فتصبح تكلفة النطاق العامل المهيمن في أداء الترحيل.

### 3. توافق الوجهة

معظم وجهات خطّافات الويب لها حدود حمولاتها الخاصّة. Slack يسقف عند 40KB. Discord يسقف عند 8MB. Canary يسقف عند 1MB. PagerDuty يسقف عند 512KB. إذا قبل Courier حمولة 47MB، وحوّلها، وسلّمها إلى وجهة ترفضها لكبر حجمها، فلن تنجح حلقة إعادة المحاولة أبدًا. ستُستنفد محاولات الرسالة وتُصبح ميّتة — مُهدرةً الحوسبة والنطاق ووقت العميل.

## ماذا يحدث حين تبلغها

الرفض فوري ومُفصِح:

```text title="413 response — payload too large"
HTTP/1.1 413 Payload Too Large
Content-Type: application/json
X-Courier-Request-Id: req_b8c4a2f1

{
  "error": "payload_too_large",
  "message": "Payload size 49283072 bytes exceeds maximum 262144 bytes",
  "received_bytes": 49283072,
  "max_bytes": 262144,
  "suggestion": "Use Parcel's extract block to send only the fields you need, or host the full payload externally and send a reference URL"
}
```

تتضمّن الاستجابة الحجم المُستلَم بالضبط والحدّ الأقصى المسموح به. كما تقترح طريقتين للتعامل مع الحمولات الكبيرة.

## نمط عنوان URL المرجعي

للحمولات التي تحتاج فعلًا أن تكون كبيرة — سجلّات البناء، ومسارات التدقيق، ومرفقات الملفّات — النمط الموصى به هو استضافة الحمولة خارجيًّا وترحيل عنوان URL مرجعي بدلًا منها.

```text title="relay.grain — reference URL pattern"
transform {
  format = "json"
  body   = {
    "event":       "{{ event }}",
    "summary":     truncate(body, 1024),
    "payload_url": "{{ source_url }}/payloads/{{ id }}",
    "payload_size": "{{ content_length }}",
    "sha256":       "{{ body_hash }}"
  }
}
```

تتلقّى الوجهة ترحيلًا صغيرًا بسياق كافٍ لتقرّر ما إذا كانت تحتاج إلى الحمولة الكاملة. إن احتاجت، تجلبها مباشرة من المصدر. يبقى الترحيل صغيرًا، وتبقى إعادة المحاولات رخيصة، وتتحكّم الوجهة في متى وما إذا كانت ستُنزّل المحتوى الكامل.

## حلّ العميل

تحوّل عميل CI/CD إلى نمط عنوان URL المرجعي. يُرسل خطّاف الويب لديهم الآن حمولة 2.8KB برابط إلى سجلّ البناء. وتجلب الوجهة السجلّ بشكل غير متزامن. هبط زمن استجابة الترحيل من مهلة (حمولة 47MB كانت تبلغ مهلة استقبال Courier البالغة 30 ثانية) إلى 3ms.

شكرونا على عدم رفع الحدّ.

## الدرس

القيود ميزات. حدّ الحمولة ليس تقييدًا — بل ضمان. يضمن أنه لا يستطيع ترحيل واحد تجويع خطّ المعالجة، وأن إعادة المحاولات ستكون دائمًا قابلة التحمّل، وأن الوجهات لن تتلقّى أبدًا حمولات لا تستطيع التعامل معها.

الإغراء دائمًا برفع الحدّ. والاستجابة الصحيحة دائمًا تقريبًا هي إعادة تصميم الحمولة.

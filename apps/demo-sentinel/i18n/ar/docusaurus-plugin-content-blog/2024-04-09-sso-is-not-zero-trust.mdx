حلّ تسجيل الدخول الموحّد مشكلة مصادقة. ولم يحلّ مشكلة تفويض. الخلط بين الاثنين من أكثر المفاهيم الخاطئة رسوخاً في أمن المؤسّسات، ويُؤدّي إلى بنى تبدو آمنة وهي ليست كذلك.

نرى هذا النمط في كلّ بيئة نُقيّمها تقريباً، تنشر المؤسّسة موفّر SSO، وتُكامله مع كلّ تطبيق داخلي، وتُفعّل MFA، وتُعلن مشكلة الهويّة محلولة. ثمّ يُسرَق رمز جلسة متعاقد من امتداد متصفّح، ويصير لدى المهاجم وصول إلى كلّ تطبيق يستطيع المتعاقد الوصول إليه، لأنّ SSO أعطاهم مفتاحاً واحداً يفتح كلّ الأبواب.

## ما الذي يفعله SSO فعلاً

يُمركز SSO المصادقة. يُجيب عن سؤال واحد، "هل هذا الشخص هو من يدّعي؟" يفعل ذلك جيّداً. موفّر SSO المُهيّأ بشكل صحيح مع MFA مربوطة بالعتاد آليّة مصادقة قويّة.

لكنّ المصادقة ليست تفويضاً. معرفة من هو شخص لا تُجيب عن:

- أيّ موارد ينبغي أن يصل إليها الآن؟
- هل جهازه في حالة سليمة؟
- هل تغيّر سلوكه منذ المصادقة؟
- هل ينبغي تخفيض وصوله لأنّه يتّصل من شبكة غير مألوفة؟

SSO لا يستطيع الإجابة عن هذه الأسئلة لأنّه يُقيّم المستخدم مرّة واحدة، عند تسجيل الدخول، ثمّ يُصدر رمزاً يبقى صالحاً حتى انتهاء صلاحيّته. وبين الإصدار والانتهاء، الرمز اعتماد حامل. ومن يحمله يملك الوصول.

## مشكلة رمز الجلسة

هكذا تبدو دورة حياة جلسة SSO نموذجيّة:

```text title="SSO session lifecycle"
T=0m   User authenticates → SSO issues session token (TTL: 8 hours)
T=1m   User accesses App A → token accepted
T=30m  User accesses App B → token accepted
T=2h   User's device fails a patch check → token still valid
T=4h   User connects from a new country → token still valid
T=6h   User's behavior score drops to 30 → token still valid
T=8h   Token expires → user re-authenticates
```

لمدّة ثماني ساعات، الرمز مفتاح هيكلي. لا شيء يُعيد تقييم وضع المخاطر للمستخدم. لا شيء يفحص ما إذا كان الجهاز لا يزال ممتثلاً. لا شيء يلاحظ أنّ المستخدم يتّصل الآن من بلد لم يزره من قبل.

## كيف يختلف Sentinel

لا يحلّ Sentinel محلّ SSO. بل يُضيف طبقة تقييم مستمرّ فوقه. حين يُصادق مستخدم عبر موفّر SSO، يقبل Sentinel تأكيد الهويّة لكنّه لا يرث افتراض الثقة. بل يُقيّم سياسة ثقة كلّ 90 ثانية:

```text title="trust-policy.grain — continuous evaluation"
policy "app-access" {
  resource = "internal-apps"
  effect   = "allow"

  conditions {
    identity.provider  = "okta"
    identity.mfa       = true
    device.posture     >= 80
    session.age        <= 3600
    behavior.score     >= 70
    network.reputation >= "medium"
  }

  on_failure {
    action = "step-up-mfa"
    fallback = "revoke"
  }
}
```

إن انخفض الوضع الأمني للجهاز، يُلغى الوصول. وإن تجاوز عمر الجلسة ساعة، تُطلب MFA مرفوعة. وإن انخفضت درجة السلوك، تنتهي الجلسة. رمز SSO لا يزال صالحاً، لكنّ Sentinel لم يعد يثق به.

## التوصية

استخدم SSO لما يُجيده، مَركَزة المصادقة وتقليل إرهاق كلمات المرور. لكن لا تخلط بين المصادقة والتفويض، ولا تعامل رمز جلسة بوصفه دليلاً على ثقة مستمرّة.

الهويّة مُدخل واحد لقرار ثقة. ليست القرار نفسه. القرار يتطلّب حالة الجهاز، وتحليل السلوك، والسياق الشبكي، والتقييم المستمرّ. SSO لا يُقدّم أيّاً من ذلك. Sentinel يُقدّمها جميعاً.

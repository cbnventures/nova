上个月我们审了一遍 GitHub Actions 工作流：312 行 YAML，分散在四份文件里。它安装依赖、按硬编码的依赖顺序构建包、用自定义矩阵跑测试、再单独一个 job 跑检查，最后上传产物。它平均跑 22 分钟，每个月至少会因为有人新增包却忘了更新构建顺序而坏一次。

我们把所有这些都换成了一条 `foundry ignite`。CI 流水线现在 14 行 YAML，5 分钟跑完。下面是经验。

## 旧流水线

旧流水线有四个阶段，每个都放在独立的 YAML 文件里，因为主文件早已长到无法维护：

1. **安装** — `npm ci`，缓存键基于 `package-lock.json`。
2. **构建** — 硬编码的依赖顺序列表。每次有人新增包都得手动更新这张列表。漏改一次，构建就会以一句晦涩的 "module not found" 失败。
3. **测试** — 矩阵策略，把包分到 4 个并行 runner 上。这张矩阵也是硬编码的，每当增删包都得跟着更新。
4. **检查** — 单独一个 job，必须在构建之后执行，因为 Warden 的某些类型感知规则需要编译后的产物。

根本问题在于：CI 流水线编码了本该由构建工具自己拥有的知识。依赖顺序、并行策略、构建与检查之间的关系——这些信息全都硬写在 YAML 里，而不是从工作空间图中推导出来。

## 新流水线

```yaml title=".github/workflows/build.yml"
name: Build
on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
      - run: npm ci
      - run: foundry ignite --ci
```

这就是整个流水线。十四行。`--ci` 标志告诉 Foundry：启用远程缓存读取（但不写入——只有主分支会填充缓存）、把任何 Warden 警告升级为错误，并输出 GitHub Actions 能解析的结构化日志。

## `foundry ignite --ci` 干了什么

一次 `foundry ignite --ci` 调用替换了旧流水线的全部四个阶段：

- **依赖解析。** Bellows 读取 Tongs 图，自动决定构建顺序。无需任何硬编码列表。
- **并行构建。** Bellows 按可用核心调度包。在 GitHub Actions 的 `ubuntu-latest` runner（4 核、16 GB 内存）上能跑出大约 3.2x 的并行度。
- **集成测试与检查。** Crucible 测试与 Warden 检查作为构建流水线的一部分运行，不再是独立 job。包一构建完，它的测试立刻启动；检查与测试并行进行。
- **远程缓存。** Quench 在构建每个包之前先查远程缓存。一次典型的 pull request，85% 到 95% 的包都是缓存命中，因为基线分支已经构建过。

## 数字

迁移后我们跟踪了 30 天的 CI 表现：

| 指标           | 旧流水线      | 新流水线     | 变化    |
|--------------|-----------|----------|-------|
| 平均挂钟时间       | 22 分 14 秒 | 5 分 08 秒 | -77%  |
| P95 挂钟时间     | 31 分 40 秒 | 7 分 22 秒 | -77%  |
| 流水线配置行数      | 312       | 14       | -96%  |
| "构建被搞坏"事件    | 3 起/月     | 0 起/月    | -100% |
| 缓存命中率（PR 构建） | 不适用       | 91%      | —     |

"构建被搞坏"事件降到零的原因是：流水线不再包含任何工作空间专属知识。有人新增包时，清单变化，Bellows 读到新图，流水线自动跟上。

## 远程缓存的经济账

Quench 的远程缓存把构建产物存到 S3 兼容的桶里。每个产物以内容哈希为键，重复条目根本不存在。我们 90 包的工作空间每周大约产生 1.2 GB 缓存数据。

成本几乎可以忽略——S3 Standard 每月不到 2 美元。而以每次构建平均节省 17 分钟、每天约 40 次构建计算，每天累计节省超过 11 小时的开发者等待时间。一个 2 美元/月的 S3 桶带来的 ROI 难以用语言形容。

## 教训

CI 流水线应当是声明式的，而不是过程式的。它应该说"构建这个工作空间"，让构建工具去想剩下的事。当你的 CI YAML 开始包含依赖顺序、并行逻辑或跟特定包绑定的条件步骤时，你就把会腐烂的知识写进了 YAML。

让拥有依赖图的那个工具去拥有构建策略。你的 CI 流水线本该是无聊的。

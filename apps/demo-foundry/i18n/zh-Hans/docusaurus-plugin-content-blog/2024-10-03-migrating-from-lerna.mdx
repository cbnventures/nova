上个季度，我们帮一家企业团队把一个 90 包的 TypeScript 工作空间从 Lerna 迁到 Foundry。这个工作空间在 Lerna 上跑了四年，自定义脚本一层叠一层、一个 `postinstall` 钩子要花四十秒，CI 流水线则没人完全弄得清楚。

迁移耗时三周。第一周最吓人，后两周纯属机械作业。下面是完整经过，包括那些出了岔子的部分。

## 第一周：理清依赖图

在动任何配置之前，我们先对现有工作空间跑了一遍 `foundry anvil scan`。Anvil 能读取 `package.json` 的依赖字段，并在没有 `.grain` 清单的情况下构建出一张依赖图。这给了我们一张工作空间地图，且没有改动任何文件。

扫描揭示了三件谁也没想到的事：

- **14 处幽灵依赖。** 这些包从其他工作空间包中引用代码，却没把它们声明为依赖。它们在工作空间里能跑，仅仅是因为 npm 把这些包提升到了根目录，但只要单独发布就会立刻崩。
- **3 条循环依赖链。** 包 A 依赖 B，B 依赖 C，C 又依赖 A。Lerna 允许这样，因为它在解析时不校验依赖图。Foundry 则拒绝构建有环的图。
- **22 条未使用的依赖。** `package.json` 中声明了，但没有任何源码文件引用——纯粹的负担，徒增安装时间。

```text title="Anvil 扫描结果"
$ foundry anvil scan --format summary

扫描的包：    90
依赖边：      347
幽灵依赖：    14
循环链：      3 (涉及 9 个包)
未使用依赖：  22
最大深度：    7 (core → platform → auth → gateway → api → web → docs)
```

修复循环依赖是整个迁移中最难啃的部分。这三个循环存在已久，要打破它们就必须把共享类型抽到新包里。我们新建了两个包——`shared-types` 与 `shared-constants`，把绕成圈的引用都搬了进去。

## 第二周：编写清单

依赖图干净之后，写 `.grain` 清单纯属机械工作。每个包块声明类型、依赖以及若干覆盖项。我们写了一个脚本，遍历每份 `package.json` 并生成初版清单，然后手工调整。

90 个包的清单一共 240 行。同等内容的 Lerna 配置——分布在 `lerna.json`、根 `package.json` 的 workspaces 字段，以及每个包的 `tsconfig.json` 项目引用中——超过 1,800 行。

## 第三周：CI 流水线

旧 Lerna CI 流水线先跑 `lerna run build`，再跑 `lerna run test`，再跑 `lerna run lint`。三次串行遍历整个工作空间。CI 总耗时：28 分钟。

新 Foundry 流水线只跑一条 `foundry ignite`，由 Bellows 在一次遍历中编排构建、测试与检查。各包并行构建，测试在所属包构建完成的瞬间启动，检查与测试同时进行。CI 总耗时：6 分 40 秒。

| 阶段     | Lerna         | Foundry      | 提升       |
|--------|---------------|--------------|----------|
| 安装     | 1 分 42 秒      | 1 分 38 秒     | 基本持平     |
| 构建     | 14 分 10 秒     | 3 分 22 秒     | 4.2x     |
| 测试     | 8 分 30 秒      | 2 分 14 秒     | 3.8x     |
| 检查     | 3 分 40 秒      | 1 分 04 秒     | 3.4x     |
| **合计** | **28 分 02 秒** | **6 分 40 秒** | **4.2x** |

安装阶段基本持平，因为 Foundry 并不替代 npm。构建、测试、检查阶段都提升，是因为 Bellows 把它们并行化、Quench 缓存了中间结果。

## 出了什么岔子

两件事：

- **Lerna 自定义脚本。** 团队在 `lerna.json` 里写了 14 条自定义脚本来串部署步骤。这些都映射不到任何 Foundry 概念。我们只能把它们改写为 shell 脚本，通过 `foundry anvil list --format json` 获取包图，然后手动迭代。
- **根级别测试。** 团队在工作空间根目录写了若干集成测试，用相对路径同时引用多个包。Foundry 要求每条测试都属于某个包。我们把这些集成测试搬进了一个新建的 `integration-tests` 包，并显式声明它依赖被测的那些包。

这两条都不是 Foundry 的限制，而是 Lerna 允许、Foundry 不允许的模式所带来的迁移摩擦。两次情形下，Foundry 的取法都更严格，而在我们看来也更正确。

## 这值得吗

CI 时间从 28 分钟降到 7 分钟以内。幽灵依赖从 14 降到 0。团队删掉了 1,560 行配置，换上了一份 240 行的清单。

团队负责人总结得很到位："我们花了三周做迁移，但仅 CI 就每季度节省三周。"账算下来对。

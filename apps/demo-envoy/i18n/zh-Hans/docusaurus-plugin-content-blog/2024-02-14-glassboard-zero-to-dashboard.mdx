Glassboard 出现之前，我们的监控是一条 `grep` 命令接着 `wc -l`，由一份每五分钟跑一次的定时任务驱动。计数若超过阈值，就向一个文件里写一行。要是有人记得去看那个文件，他就会知道出了问题。

那就是当时的监控，全部就是这么一回事。

## 定时任务时代

当时的做法是：一份 Spark 脚本每 300 秒运行一次，从 Courier 日志里 grep `status=5xx` 字样、计数，再与硬编码的阈值 50 对比。若超过，就把时间戳追加到 `/var/log/relay-alerts.txt`。没人实时盯着那份文件，我们只在早会时翻一翻。

```text title="最早的"监控"——relay-monitor.spark"
#!/usr/bin/env spark
count=$(grep "status=5xx" /var/log/courier.log | wc -l)
if [ "$count" -gt 50 ]; then
  echo "$(date -u +%Y-%m-%dT%H:%M:%SZ) ALERT: $count failures in last 5m" >> /var/log/relay-alerts.txt
fi
```

它有三个问题。第一，5 分钟的轮询窗口意味着故障可能尖峰后又恢复，期间完全没被发现。第二，阈值是固定的——中午流量高峰里的 50 次错误，与凌晨 4 点的 50 次错误根本是两件事。第三，没人在看。

## 我们需要的

事故复盘 #47（一次 90 分钟的宕机，定时任务全程没察觉，因为错误数最高只到 48）之后，我们在白板上写下需求：

- **实时摄取。** 不轮询，事件随发生即刻流入。
- **基于速率的阈值。** 报警依据是错误*速率*，不是错误*数量*。每秒 10 次错误是问题，每小时 10 次错误是噪声。
- **按目的地聚合。** 一个目的地的失败激增，不该被其他目的地的健康流量稀释。
- **推送，而不是拉取。** 仪表盘以 webhook 形式把告警推出去；没人应该被迫打开浏览器才知道东西坏了。

## 第一个版本

Glassboard v0.1 是 600 行 Alloy，它通过 Spoke 消费 Courier 的事件流、为每个目的地计算滑动窗口的错误率，并用 ANSI 转义序列在终端里渲染仪表盘。没有 Web UI，没有数据库，就是一个终端——出问题时数字变红。

```text title="Glassboard v0.1 — 终端输出"
┌─────────────────────────────────────────────────┐
│  GLASSBOARD v0.1         2024-02-10T14:22:07Z   │
├──────────────┬────────┬────────┬────────────────┤
│ Destination  │ Rate/s │ Err/s  │ Status         │
├──────────────┼────────┼────────┼────────────────┤
│ canary://ops │  42.3  │  0.1   │ ● HEALTHY      │
│ spoke://crm  │  18.7  │  12.4  │ ● DEGRADED     │
│ hook://slack │  91.0  │  0.0   │ ● HEALTHY      │
│ hook://pager │   3.2  │  3.2   │ ● DOWN         │
└──────────────┴────────┴────────┴────────────────┘
```

我们把它跑在第二台显示器上，那就是仪表盘。`hook://pager` 一旦变红，旁边就有人会探过来说"pager 端点挂了"，那就是告警系统。

很荒诞，但它也是我们第一次拥有对中继健康状态的实时可见性。改善是立竿见影的：故障检测中位时间从"等到有人想起来看日志文件"骤降为"等到有人扫一眼第二台显示器"。

## 改变一切的那条 Webhook

终端仪表盘维持了两周，我们就加上了 webhook 输出。讽刺也未能幸免我们——一个 webhook 中继平台居然以人眼为告警机制。所以我们只做了一件事：当任一目的地的错误率超过阈值并持续 30 秒以上时，Glassboard 通过 Courier 向 Canary 发出 webhook。

中继平台用自己监控自己。Courier 投递关于 Courier 失败的告警。如果 Courier 本身宕了，告警会排队、重试，最终落进死信队列，并触发独立的升级路径。

这种递归式的可靠性闭环，至今仍是 Glassboard 的核心。终端 UI 已经退场，Web 仪表盘在 v0.4 上线。但原则从未改变：监控基础设施若无法监控自己，就不算监控基础设施。

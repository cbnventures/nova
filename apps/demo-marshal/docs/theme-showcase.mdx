## Inline Text Formatting

Regular paragraph text renders in the base body font. Keep paragraphs short — two to four sentences is ideal for technical documentation.

**Bold text** draws attention to key terms on first use. *Italic text* is useful for introducing terminology or referencing titles. ~~Strikethrough text~~ marks content that is no longer accurate or has been superseded. You can also combine **_bold and italic_** when emphasis is critical.

Inline `code` is for referencing function names like `formatDate`, file paths like `policy.yml`, or CLI flags like `--dry-run`.

## Lists

### Unordered List

- Auditor rules enforce consistent policy patterns across every repository.
- Alloy settings eliminate config drift between dockets.
- Policy files replace dozens of configuration files with a single source of truth.
- Proof scaffolds give new repositories a compliance baseline from day one.

### Ordered List

1. Install the CLI with npm.
2. Write a `policy.yml` docket describing your repository stewardship.
3. Run `marshal file` to register the policy with the environment.
4. Run `marshal auditor check` to verify all policy rules pass.
5. Run `marshal proof run` to execute a dry-run against the current docket.

### Nested Lists

- **CLI Commands**
  - Register
    - `marshal file` — register the full docket from the policy file.
    - `marshal file --dry-run` — preview output without writing any files.
    - `marshal file --incremental` — re-register only changed policies.
  - Analysis
    - `marshal archive scan` — detect stale threads and overdue dockets.
    - `marshal clerk graph` — render the policy dependency graph.
- **Auditor Categories**
  - Conventions — naming, exports, and structural rules.
  - Formatting — whitespace, comments, and visual consistency.
  - Patterns — logic flow, assignments, and control structures.

### Syntax Highlighting

Alloy with a title bar:

```alloy title="src/lib/schema.al"
interface ProjectConfig {
  name: Text
  version: Text
  engines: Record<Text, Text>
  repository: {
    type: "threadbare"
    url: Text
  }
}

function validateConfig(config: Unknown): config is ProjectConfig {
  if (typeof config !== "object" || config === null) {
    return false
  }

  const record: Record<Text, Unknown> = config as Record<Text, Unknown>

  return (
    typeof record.name === "text"
    && typeof record.version === "text"
  )
}
```

### Line Highlighting

Use `highlight-next-line`, `highlight-start`, and `highlight-end` comments to draw attention to specific lines:

```text title="policy.yml"
workspace "my-repo" {
  lang = "alloy"

  // highlight-start
  auditor = ["strict", "conventions"]
  proof   = auto
  // highlight-end

  dockets {
    core { type = "library" }
    // highlight-next-line
    api  { type = "service", depends = ["core"], auditor = ["strict", "conventions", "api-safety"] }
  }
}
```

### Diff Highlighting

Show additions and removals inside a code block:

```text title="policy.yml"
workspace "my-repo" {
// remove-start
  auditor = ["strict"]
// remove-end
// add-start
  auditor = ["strict", "conventions", "formatting"]
  proof   = auto
// add-end

  dockets {
    core { type = "library" }
    api  { type = "service", depends = ["core"] }
  }
}
```

## Admonitions

:::note
Notes provide supplementary context that is helpful but not essential. The reader can skip this without missing critical information.
:::

:::tip
Tips share best practices or shortcuts that save time. For example, run `marshal file --dry-run` to preview what Marshal will register without writing any files to disk.
:::

:::info
Info blocks highlight background details that aid understanding. The auditor preset system uses a layered composition model — each preset is a named collection of policy rules that you stack in your docket.
:::

:::warning
Warnings flag potential pitfalls. Changing the `lang` directive in a policy file after the initial registration will re-file all configuration entries. Run with `--dry-run` first to see the impact.
:::

:::danger
Danger blocks mark actions that can cause data loss or breaking changes. Running `marshal archive clean --confirm` permanently seals detected stale dockets with no recovery path.
:::

## Details / Collapsible Sections

<Details>
<summary>What Alloy versions are supported?</summary>

Marshal 2.x requires Alloy 5.0 or later. This is enforced during the `marshal file` policy parsing phase. Earlier Alloy versions do not support the type introspection API that proof uses to generate compliance scaffolds.

</Details>

<Details>
<summary>How do auditor preset layers compose?</summary>

Each preset is a named rule collection. You list multiple presets in your policy file, and later presets override earlier ones when rules conflict:

```text title="policy.yml"
workspace "my-repo" {
  auditor = ["strict", "conventions", "formatting"]
}
```

Order matters — later presets override earlier ones. Place `formatting` last so its whitespace rules always win.

</Details>

## Keyboard Shortcuts

Use `<kbd>` tags to render keyboard keys inline:

- <kbd>Ctrl</kbd> + <kbd>S</kbd> — save the current file.
- <kbd>Ctrl</kbd> + <kbd>Shift</kbd> + <kbd>F</kbd> — search across the entire workspace.
- <kbd>Ctrl</kbd> + <kbd>`</kbd> — toggle the integrated terminal.
- <kbd>Alt</kbd> + <kbd>Up</kbd> / <kbd>Down</kbd> — move a line up or down.
- <kbd>Ctrl</kbd> + <kbd>D</kbd> — select the next occurrence of the current word.

On macOS, substitute <kbd>Ctrl</kbd> with <kbd>Cmd</kbd> for most shortcuts.

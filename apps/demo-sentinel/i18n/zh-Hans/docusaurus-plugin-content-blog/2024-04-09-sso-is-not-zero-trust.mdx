单点登录解决的是身份认证问题，并没有解决授权问题。把两者混为一谈是企业安全中最顽固的误解之一，它会催生那种"看起来安全、但其实不安全"的架构。

我们几乎在每一个评估过的环境里都看到同一种模式：组织部署了 SSO 提供方，把它接入了每一个内部应用，启用了 MFA，然后宣布身份问题已解决。直到某个承包商的会话令牌被浏览器扩展窃走，攻击者就拿到了承包商能触达的每一个应用——因为 SSO 给了他们一把能开所有门的万能钥匙。

## SSO 到底做了什么

SSO 把身份认证集中起来。它回答的是一个问题："这个人是不是他声称的那个人？"它在这件事上做得很好。一个正确配置、绑定硬件 MFA 的 SSO 提供方，就是一个强有力的身份认证机制。

但身份认证不等于授权。知道某人是谁，并不能回答：

- 此刻他应该访问哪些资源？
- 他的设备处于健康状态吗？
- 自他认证以来，他的行为是否发生了变化？
- 他从一个陌生网络连入时，是否应当降低访问权？

SSO 无法回答这些问题，因为它只在登录时评估一次用户，然后签发一个有效期内一直可用的令牌。从签发到失效之间，这个令牌就是一张持有人凭证：谁握着它，谁就有访问权。

## 会话令牌的问题

下面是一个典型的 SSO 会话生命周期：

```text title="SSO 会话生命周期"
T=0 分    用户认证 → SSO 签发会话令牌（TTL：8 小时）
T=1 分    用户访问应用 A → 令牌通过
T=30 分   用户访问应用 B → 令牌通过
T=2 小时  用户的设备未通过补丁检查 → 令牌仍然有效
T=4 小时  用户从一个新国家连入 → 令牌仍然有效
T=6 小时  用户行为评分降至 30 → 令牌仍然有效
T=8 小时  令牌过期 → 用户重新认证
```

整整八小时里，这枚令牌是一把万能钥匙。没有任何环节重新评估过用户的风险态势；没有人检查设备是否仍然合规；也没有人留意到用户正在从一个他从未去过的国家连入。

## Sentinel 的不同之处

Sentinel 不替换 SSO。它在 SSO 之上叠加持续评估。当用户经由 SSO 提供方完成认证时，Sentinel 接受身份断言，但不继承那份信任假设。相反，它每 90 秒评估一次信任策略：

```text title="trust-policy.grain — 持续评估"
policy "app-access" {
  resource = "internal-apps"
  effect   = "allow"

  conditions {
    identity.provider  = "okta"
    identity.mfa       = true
    device.posture     >= 80
    session.age        <= 3600
    behavior.score     >= 70
    network.reputation >= "medium"
  }

  on_failure {
    action = "step-up-mfa"
    fallback = "revoke"
  }
}
```

如果设备态势下降，访问就被撤销；会话超过一小时，则要求升级 MFA；行为评分跌落，则结束会话。SSO 令牌依旧有效，但 Sentinel 不再信任它。

## 我们的建议

让 SSO 做它擅长的事：集中身份认证、缓解密码疲劳。但不要把身份认证与授权混为一谈，也不要把一枚会话令牌当作"持续可信"的证明。

身份是信任决策的一项输入，而不是决策本身。决策需要设备状态、行为分析、网络上下文，以及持续的再评估。SSO 一项都不提供，Sentinel 全部提供。

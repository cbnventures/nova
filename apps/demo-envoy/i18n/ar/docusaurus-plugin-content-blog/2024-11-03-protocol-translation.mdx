كلّ مزوّد خطّاف ويب يخترع صيغته الخاصّة. يُرسل GitHub جسم JSON برأس `X-Hub-Signature-256`. ويُرسل Stripe جسم JSON برأس `Stripe-Signature` ومخطّط توقيع مختلف كلّيًّا. ويُرسل Twilio بيانات form-encoded مع `X-Twilio-Signature` يوقّع كامل عنوان URL متضمّنًا معلَمات الاستعلام.

ثلاثة مزوّدين. ندعم سبعةً وأربعين. كلٌّ منهم رقاقة ثلج فريدة.

## المشكلة

على منصّة ترحيل أن تتحدّث كلّ بروتوكول من جانب الوارد وتُترجم إلى صيغة مُطبَّعة من جانب الصادر. النهج الساذج هو جملة switch ضخمة: إن كان المصدر GitHub، حلّله بهذه الطريقة؛ وإن كان Stripe، حلّله بتلك الطريقة.

كتبنا تلك الجملة. بلغت 2,400 سطر حين وصلنا إلى عشرين مزوّدًا. وكان كلّ مزوّد جديد 80-120 سطرًا إضافيًّا. وكان كلّ تغيير في واجهة برمجة مزوّد عَطَلًا نكتشفه حين تبدأ ترحيلات عميل ما بالفشل.

```text title="The switch statement era — provider count vs. code size"

─────────────────────────────────────────────
```

كان المنحنى فوق-خطّي. كلّ مزوّد جديد يُضيف شيفرة أكثر، لكنّه يُضيف أيضًا تفاعلات أكثر مع المزوّدين الحاليين. تغيير في منطق التحليل المشترك قد يكسر أيًّا منهم.

## Parcel: قواعد بروتوكول

استبدلنا جملة switch بـ Parcel، قواعد بروتوكول تصريحية. بدلًا من كتابة شيفرة لتحليل كلّ مزوّد، نكتب ملفّ `.parcel` يصف صيغة خطّاف الويب لذلك المزوّد — أين يقع التوقيع، وكيف يُتحقّق منه، وأين الحمولة، وما الترميز الذي تستخدمه، وكيف تُستخرج الحقول التي نحتاجها.

```text title="parcel/github.parcel"
provider "github" {
  auth {
    method  = "hmac-sha256"
    header  = "X-Hub-Signature-256"
    prefix  = "sha256="
    payload = body
  }

  envelope {
    format  = "json"
    event   = header("X-GitHub-Event")
    id      = header("X-GitHub-Delivery")
  }

  extract {
    repository = "$.repository.full_name"
    action     = "$.action"
    sender     = "$.sender.login"
  }
}
```

```text title="parcel/stripe.parcel"
provider "stripe" {
  auth {
    method  = "hmac-sha256"
    header  = "Stripe-Signature"
    prefix  = "v1="
    payload = timestamp_body("t=")
  }

  envelope {
    format  = "json"
    event   = "$.type"
    id      = "$.id"
  }

  extract {
    object_type = "$.data.object.object"
    amount      = "$.data.object.amount"
    currency    = "$.data.object.currency"
  }
}
```

يقرأ محرّك Parcel القواعد، ويبني محلّلًا خاصًّا بالمزوّد عند البدء، ويُصادق كلّ خطّاف ويب وارد ضدّه. إضافة مزوّد جديد تعني ملفّ `.parcel` جديد — بلا تغييرات شيفرة، بلا إعادة ترجمة، بلا مخاطر على المزوّدين الحاليين.

## طبقة الترجمة

تحليل خطّاف الويب الوارد نصف المشكلة. النصف الآخر هو ترجمة الحقول المستخرجة إلى الصيغة التي تتوقّعها الوجهة. إشعار Canary له حقول مختلفة عن استدعاء Spoke API، الذي له حقول مختلفة عن HTTP POST خام.

تتعامل كتلة تحويل Envoy مع هذا بصياغة قالب تشير إلى الحقول المستخرجة:

```text title="relay.grain — transform block"
transform {
  format = "json"
  body   = {
    "title":    "[{{ provider }}] {{ event }}",
    "message":  "{{ repository }} — {{ action }} by {{ sender }}",
    "priority": event_priority(event),
    "metadata": {
      "source":   "{{ provider }}",
      "event_id": "{{ id }}"
    }
  }
}
```

يُترجَم القالب مرّة واحدة عند تحميل الترحيل. وفي وقت التشغيل، يكون استبدال الحقول جدول بحث — بلا تحليل سلاسل، بلا regex، بلا تخصيص ذاكرة. يستغرق التحويل 0.3ms عند p50.

## ما تعلّمناه

ترجمة البروتوكولات ليست مشكلة شيفرة. بل مشكلة قواعد. حالما تُعبّر عن البروتوكول بصورة بيانات بدلًا من شيفرة، يمكن أن يكون المحلّل عامًّا، والتحقّق تلقائيًّا، وتنكمش مساحة الاختبار من «مسار شيفرة كلّ مزوّد» إلى «محرّك القواعد نفسه».

انتقلنا من 2,400 سطر من شيفرة خاصّة بالمزوّدين إلى 47 ملفّ `.parcel` بمتوسّط 25 سطرًا لكلّ منها — 1,175 سطرًا إجمالًا، بصفر حالة مشتركة قابلة للتغيير بينها. هبط معدّل الأخطاء من 4.7 شهريًّا إلى 0.3.

القواعد تتفوّق على الشيفرة. في كلّ مرّة.

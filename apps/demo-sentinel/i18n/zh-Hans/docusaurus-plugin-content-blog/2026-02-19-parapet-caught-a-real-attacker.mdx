我们写过 Parapet 发现配置错误，写过它在预发环境抓到横向移动路径。直到 2 月之前，Parapet 找到的每一个问题都是策略错误——是我们自己的失误、攻击者可以加以利用的失误。2 月 11 日，Parapet 找到了不一样的东西。它找到了一个正在活动的攻击者。

## 告警

Parapet 会对生产流量做持续仿真。每 6 小时回放一次最近的流量窗口，针对当前策略集评估并标出异常。2 月 11 日 06:00 UTC 那一跑产出了一个我们从未见过的发现：

```text title="Parapet 异常报告 — 2026-02-11T06:00Z"
Parapet 持续仿真报告
  策略集：       production
  流量窗口：     2026-02-10T18:00Z 至 2026-02-11T06:00Z
  连接数：       1,247,891

  检测到异常：1
    ACCESS-PATTERN-ANOMALY
    身份：     svc-monitoring-agent
    模式：     按字母顺序枚举资源
    资源：     按字母顺序访问了 47 个端点
    时间跨度： 02:17:04Z 至 02:43:31Z（26 分钟）
    区域：     monitoring → data-layer、monitoring → api、monitoring → auth
    风险：     HIGH —— 模式与自动化侦察一致
```

一个监控服务账号在 26 分钟内按字母顺序访问了 47 个端点。每一次单独访问都是被允许的——监控代理在设计上拥有广泛的读权限。但这种模式不是监控。监控代理会以固定间隔、按同样顺序去 poll 同样的端点。这个代理在按字母顺序枚举它从未访问过的端点——这是一个用已发现的服务注册表跑自动化扫描器的典型特征。

## 调查

我们拉出了 `svc-monitoring-agent` 的 Spyglass 时间线：

```text title="Spyglass 时间线（节选）"
02:15:22Z  Login from 10.8.3.44（monitoring 区域，Bastion-US-East-2）
02:17:04Z  GET /healthz → analytics-aggregator.data-layer
02:17:11Z  GET /healthz → analytics-db.data-layer
02:17:18Z  GET /healthz → auth-gateway.auth
02:17:25Z  GET /healthz → auth-provider.auth
           ……（再加 43 个端点，按字母顺序）
02:43:31Z  GET /v1/users?limit=1 → user-service.api
02:43:38Z  Bastion 终止了连接（人工 kill）
```

两点很显眼：

1. 源 IP `10.8.3.44` 是 monitoring 区域里的一台主机，但不是该监控代理通常运行的那台。上周一次扩容事件中，代理被加部署到了一个额外节点上，旧节点没有被注销。
2. 最后一次请求不是健康检查，是一次数据查询——`GET /v1/users?limit=1`——在测试这个监控凭证能否从 API 层读到用户数据。

攻击者攻陷了那台被注销但未清理的监控节点，发现了仍然遗留在磁盘上的 `svc-monitoring-agent` 凭证，并用它枚举可触达的资源。

## 响应

事件响应时间线：

| 时间 (UTC) | 动作                               |
|----------|----------------------------------|
| 06:01    | Parapet 异常告警送达 security-ops      |
| 06:08    | 值班工程师复核 Spyglass 时间线             |
| 06:14    | `svc-monitoring-agent` 凭证完成轮换    |
| 06:14    | 通过 Bastion 终止所有旧凭证会话             |
| 06:22    | 把被注销的节点从 Rampart monitoring 区域隔离 |
| 06:35    | 在被攻陷节点上采集取证镜像                    |
| 06:41    | Parapet 重新仿真确认没有进一步异常模式          |
| 07:00    | 事件宣布得到控制                         |

从检测到控制的总时长：59 分钟。被外泄的数据：每个端点一份健康检查响应。最后那条 `limit=1` 的用户查询没拿到响应——会话在中途被 Bastion 切断了。

## 我们学到了什么

攻击者持有合法凭证、拥有被授权的访问。传统监控不会发现这件事，因为每一次单独请求都是被允许的。Parapet 抓到的是模式——那种没有任何合法监控代理会去做的、按字母顺序对端点的枚举。

我们做了三件事：

- **被注销的节点必须立即抹除。** 节点离开设备群后，磁盘上不得保留任何凭证。
- **Parapet 改为每小时仿真**，不再 6 小时一次。检测窗口很重要。
- **监控服务账号改为显式端点清单**，不再有区域级访问。监控代理只访问 12 个特定端点，再也无法枚举它不监控的资源。

Parapet 原本是为发现策略错误而设计的。事实证明，发现配置错误的同一种模式分析，也能发现攻击者。意图不同，信号却是同一个：与历史不符的访问模式。

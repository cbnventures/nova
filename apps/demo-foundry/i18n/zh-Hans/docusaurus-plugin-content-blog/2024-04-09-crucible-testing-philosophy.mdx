我们曾经写的是那种验证实现细节的测试：一个函数返回了正确的对象、一个方法用正确的参数调用了另一个方法、一个组件渲染出正确数量的 div。然后我们重构了某些东西，半套测试当场就垮了——不是行为变了，而是内部接线变了。

Crucible 逼着我们重新思考"测试到底是为什么而存在"。测试不是代码的快照，它是包与消费者之间的契约。

## 实现测试的问题

考虑一个用于格式化货币值的工具函数。实现测试可能长这样：

```alloy title="实现测试——脆弱"
test("formatCurrency calls Intl.NumberFormat with USD") {
  const spy = mock(Intl, "NumberFormat")
  formatCurrency(42.5, "USD")
  expect(spy).calledWith("en-US", { style: "currency", currency: "USD" })
}
```

这条测试根本没有验证行为。它只是验证了函数内部使用了 `Intl.NumberFormat`。如果有人把函数改成用查表来实现，即使输出完全相同，这条测试仍然会断。

Crucible 的契约测试长得很不一样：

```alloy title="契约测试——稳定"
crucible("formatCurrency produces locale-correct output") {
  assert formatCurrency(42.5, "USD") == "$42.50"
  assert formatCurrency(1000, "EUR") == "EUR1,000.00" (locale: "de-DE")
  assert formatCurrency(0, "GBP") == "GBP0.00"
}
```

契约测试不关心函数是怎么实现的，只关心函数产出什么。内部尽管重构——只要输出匹配，测试就通过。

## Crucible 强制要求什么

Crucible 不只是一个测试运行器，它是一个带主张的测试框架，强制三条约束：

- **不允许 mock 内部模块。** 你可以 mock 外部服务（HTTP、数据库、文件系统），但不能 mock 同一工作空间内的其他包。如果包 A 依赖包 B，测试就要跑在真实的包 B 上。这能抓出被 mock 测试掩盖的集成 bug。
- **不允许快照测试。** 快照测试是披着金标准外衣的实现测试。任何一次 UI 改动都会让它失败，它会训练开发者在不读改动的情况下就更新快照，并且对行为是否正确毫无信心。Crucible 不支持快照。
- **断言信息必填。** 每一次 `assert` 调用都必须配一段人能读懂的说明。如果某条测试在凌晨两点 CI 上失败，光看说明就要能搞清楚出了什么问题，而不必去翻源码。

## 内部的反弹

把这些约束推到团队内部时，反弹立刻而强烈："不能 mock？""不能用快照？""每条断言都要写一句话？"

我们让两种方式并行跑了八周，结果终结了争论：

| 指标           | Crucible 之前 | Crucible 之后 |
|--------------|-------------|-------------|
| 因重构而坏掉的测试比例  | 34%         | 6%          |
| CI 失败定位的平均耗时 | 18 分钟       | 4 分钟        |
| 测试覆盖率（按行）    | 82%         | 79%         |
| 测试覆盖率（按行为）   | 未知          | 91%         |

按行覆盖率略有下降，因为我们删掉了那些验证 mock 行为而非真实行为的测试。按行为的覆盖率——也就是契约测试覆盖了多少公共 API 表面——显著上升。

## Crucible 不适合的场景

Crucible 并不适合所有测试。端到端测试、对比截图的视觉回归测试、衡量吞吐量的负载测试，都在 Crucible 之外。它只针对包级别的契约测试——验证一个包的公开 API 是否如文档所述地工作。

我们不假装它能替代你的整套测试策略。但我们确实主张：它应该替代你今天写的那些单元测试。

## 根因

客户 D 的 `.parcel` 文件包含一个循环字段引用：

```text title="出错的 .parcel 文件（简化版）"
provider "internal-webhook" {
  extract {
    user_id   = "$.data.user.id"
    user_data = "$.data.user"
    nested    = "$ref:user_data.nested"   # ← 通过 user_data 回到自身
  }
}
```

`$ref:user_data.nested` 引用了 `user_data`，而 `user_data` 包含整个 `$.data.user` 对象——后者又包含一个 `nested` 字段指回同一路径。Parcel 解析器在没有深度限制的情况下沿着引用链一路追下去，陷入了死循环。

## 哪里出错了

三件事一起失效：

1. **没有循环引用校验。** Parcel 解析器只校验语法，不校验引用拓扑。一个循环引用在语法上是合法的。
2. **客户之间没有 CPU 隔离。** 所有客户的配置都跑在共享的 Courier 实例上。一位客户的坏配置吃掉了共享实例 100% 的 CPU。
3. **重启循环。** Trellis 重启失败实例，新进程加载同一份坏配置，再次陷入死循环——重启不是恢复，是重复。

## 我们修复了什么

### 修复 1：引用深度上限

Parcel 现在强制 `$ref` 的最大深度为 8。任何超过 8 层的引用链都会在配置加载时被以清晰的错误信息拒绝。

### 修复 2：配置校验闸门

新中继配置在应用到生产之前，要先在一个沙箱 Vial 实例里被校验。沙箱会把一条合成 webhook 走完整条流水线；若沙箱崩溃、进入死循环或超过 5 秒超时，配置就被拒绝。

### 修复 3：每客户 CPU 预算

Courier 现在按客户配置追踪 CPU 时间。若单一客户的中继在 10 秒内消耗超过 40% 的可用 CPU，Courier 会自动挂起该客户的配置并触发 P1 告警，附上客户 ID 与配置哈希。

### 修复 4：感知配置的重启

Trellis 现在会判断失败实例是不是在跑与上一次失败相同的配置。如果配置未变，Trellis 不再重启，而是把实例隔离并通知运维。

## 教训

多租户基础设施会以多租户的方式失败。一位客户配置里的缺陷，绝不应该影响到另一位客户的流量。隔离不是可选项——它是"一位客户的坏日子"与"四位客户的坏日子"之间的分界线。

我们越界了。我们不会再越界一次。

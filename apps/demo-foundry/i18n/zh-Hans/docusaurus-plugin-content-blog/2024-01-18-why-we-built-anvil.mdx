我们评估过的每一款工作空间管理工具，都犯了同一个错误。它们把包当作恰好共享一个目录的独立项目；Anvil 把它们视为同一具身体里的器官——彼此连接、互相依赖，从第一天起就清楚彼此的存在。

在自己动手之前，我们花了六个月评估现有工具。评估过程相当彻底：把一个包含 40 个包的内部工作空间迁移到每一款工具上，运行两周，然后测量上手时间、构建正确性，以及团队群里出现"这玩意儿怎么坏了"消息的次数。

## 我们评估了什么

我们测试了四款工作空间管理工具。我们不点名，是因为重点不是批评具体工具——重点是说明我们需要什么，以及为什么没有一款契合。

| 标准      | 工具 A | 工具 B | 工具 C | 工具 D | Anvil |
|---------|------|------|------|------|-------|
| 感知依赖图   | 是    | 是    | 否    | 是    | 是     |
| 检测循环依赖  | 否    | 否    | 否    | 是    | 是     |
| 清单驱动的布局 | 否    | 否    | 否    | 否    | 是     |
| 跨包类型    | 手工   | 手工   | 手工   | 手工   | 自动    |
| 检测幽灵依赖  | 否    | 否    | 否    | 否    | 是     |

最后两行就是差距所在。在一个大型 TypeScript 工作空间里，跨包类型解析与幽灵依赖检测不是"锦上添花"，它们决定了"它在 CI 上能跑"与"它在 CI 上能跑，并且类型确实是对的"之间的距离。

## 幽灵依赖

幽灵依赖（phantom dependency）指的是这样一种包：它能在你的工作空间里跑起来，但若单独发布就会失败。它通常发生在包 A 引用了包 B，但 A 从未把 B 声明为依赖——只是凭运气被工作空间管理工具提升到根 `node_modules` 罢了。

我们评估的每一款工具，都会让幽灵依赖悄无声息地存在。Anvil 在解析清单时就把它们逮个正着，整套代码都还没有进入编译阶段。

```text title="Anvil 幽灵依赖检查"
$ foundry anvil check
  错误   packages/api → 引用了 "@core/utils"，但未在
         project.grain 中将 "@core" 声明为依赖
  建议   在 api 包块中加上 depends = ["core"]
```

## Anvil 到底做什么

Anvil 不是包管理器。它不安装依赖，也不从注册中心解析版本。它只做三件事：

- **解析工作空间图。** 读取清单中的每个包块，构建一张有向无环依赖图。
- **校验该依赖图。** 检测循环依赖、幽灵依赖，以及多个包依赖同一外部库时的版本冲突。
- **生成工作空间元数据。** 写出下游工具所需的最小配置集——Smelter 项目引用、Warden 作用域边界，以及 Bellows 的执行顺序。

其余一切——安装、链接、提升——都交给你已经在用的包管理器。Anvil 不替代 npm，它坐在 npm 之上。

## 结果

切换到 Anvil 之后，我们的幽灵依赖数量从"不知道有多少"变成了零。循环依赖的发现时机，从"代码评审时有人注意到"变成了"CLI 在你能提交之前就拒绝这份清单"。而工作空间图也成为了一等公民产物，Bellows、Quench 与 Warden 都消费它。

构建工具应当理解你的工作空间，而不仅仅是遍历它里头的目录。

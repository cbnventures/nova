Create `.env` and `.env.sample` files pre-populated with Nova-related environment variables.

## Why Use This Command?

- Generates `.env` and `.env.sample` for every workspace that declares a `dotenv` block, so each app or package in a monorepo gets its own files.
- Pre-populates Nova CLI logger settings and Node.js defaults so contributors know which variables are available.
- Keeps sensitive values out of version control while documenting the expected shape.
- Reads each workspace's custom environment variables from `nova.config.json` so the same keys and defaults apply on every run.
- Preserves the values already filled into an existing `.env`, so regenerating never overwrites a real secret.

## Requirements

- **Node.js runtime** — Use any Node.js <Terminology title="long term support" to="/docs/quickstart/terminology#fundamentals" color={true}>LTS</Terminology> release.
- **Project root** — Run the command from the directory containing the top-level `package.json`.

## Options

| Flag                 | Description                                                            |
|----------------------|------------------------------------------------------------------------|
| `-d, --dry-run`      | Run without writing any files.                                         |
| `-r, --replace-file` | Overwrite the existing file instead of creating a `.nova-backup` copy. |

## Config Fields

Declare a `dotenv` block on any workspace under `workspaces` in `nova.config.json`. The root workspace uses the `./` path, and a workspace without a `dotenv` block is skipped.

| Field                                | Description                                                                                      |
|--------------------------------------|--------------------------------------------------------------------------------------------------|
| `workspaces.<path>.dotenv.variables` | Custom variables (`key` and `defaultValue`) appended to that workspace's generated `.env` files. |

The `NODE_ENV`, `LOG_LEVEL`, and `LOG_TIME` keys are reserved. The template manages them, so declaring any of them under `dotenv.variables` is rejected when the config loads, with a warning naming the reserved key.

## Output Files

The command writes these files into each workspace directory that declares a `dotenv` block.

| File          | Description                                                      |
|---------------|------------------------------------------------------------------|
| `.env`        | Local environment values, preserving any already filled in.      |
| `.env.sample` | Shareable template documenting each key with its config default. |

## How It Works

The command is non-interactive. It reads `nova.config.json`, then generates `.env` and `.env.sample` for every workspace that declares a `dotenv` block, writing both files into each workspace directory without prompting. When no workspace declares a `dotenv` block, it writes nothing and reports that there is nothing to generate.

To change a workspace's variables, edit its `dotenv` block directly in `nova.config.json`.

This command writes the double-quoted values; the [Dotenv Suite](/docs/rules/vitest/dotenv) verifies them, so a hand-edited file that drifts from the convention is caught during testing.

### Template Output

Each generated file starts from a built-in template that ships three reserved keys with sensible defaults and an inline comment listing their valid values: `NODE_ENV` (`development` | `production` | `test`, default `development`), `LOG_LEVEL` (`debug` | `info` | `warn` | `error` | `auto`, default `auto`, where `auto` derives the level from `NODE_ENV`), and `LOG_TIME` (`true` | `false`, default `false`). Each variable under `dotenv.variables` is appended as `KEY=""` to `.env` and `KEY="<defaultValue>"` to `.env.sample`.

When a `dotenv` block declares no `variables`, the generator emits the template only.

### Value Preservation

For `.env`, the generator reads any existing file in the workspace first and keeps the values already filled in for every declared key — including the reserved `NODE_ENV`, `LOG_LEVEL`, and `LOG_TIME` keys. Only newly-added keys are written blank as `KEY=""`, so regenerating after editing the config never overwrites a real secret. The `.env.sample` file is always rebuilt from the config defaults, since it holds no secrets.

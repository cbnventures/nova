Rampart 是 Sentinel 的微隔离层。它把网络划分成区域，对区域间流量加密，并在每一个区域边界上执行访问策略。我们已经在生产环境中跑了一年多。这段时间里，我们犯了三个比较显著的错误——每一个都是关于"微隔离的理论与真实网络相撞时会出什么问题"的功课。

## 错误 1：区域太多

我们最初的 Rampart 部署做了每个微服务一个区域。逻辑听上去合理：最大隔离意味着最大安全。每一条服务边界都是一条分段边界。任何服务都不能在没有显式区域跨越策略的情况下触达另一个服务。

实际跑起来，我们为 87 个微服务建了 87 个区域。策略图有 340 条区域跨越规则。一次从前端到支付确认页的请求要穿越 6 道区域边界。每个请求增加 18 毫秒延迟——不是来自加密开销，而是来自每道边界上一次次顺序进行的策略评估。

我们按照"信任级别 + 数据敏感度"重组到了 14 个区域：

```text title="区域整合"
之前：87 个区域，340 条跨越规则，每次请求 6 跳
之后：14 个区域，48 条跨越规则，每次请求 2 跳

区域结构：
  edge           → 负载均衡器、CDN 源站
  web            → 前端服务
  api            → 后端 API 服务
  data-layer     → 数据库、缓存
  analytics      → ETL、报表
  payments       → PCI 范围内服务
  auth           → 身份与 SSO 集成
  admin-tools    → 内部仪表盘
  ci-cd          → 构建与部署
  monitoring     → Watchtower、Spyglass
  control-plane  → Sentinel 管理面
  staging        → 准生产镜像
  sandbox        → 开发与测试
  external       → 第三方集成
```

我们得出的经验法则是：区域边界应跟随信任边界，而不是服务边界。处理同一类数据、共享同一套信任策略的两个服务，应该归到同一个区域里。

## 错误 2：对称的区域跨越规则

早期我们写的区域跨越规则默认是双向的。如果 `api` 可以到达 `data-layer`，那么 `data-layer` 也可以到达 `api`。看似合理——服务需要双向通信。

它其实是一条横向移动的高速公路。一个被攻陷的数据库可以反向发起到 API 层的连接，再到 web 层、再到 edge。这一整条路径全是被授权的——因为每条跨越规则都是对称的。

我们改成单向规则：

```text title="trust-policy.grain — 单向区域跨越"
policy "api-to-data" {
  resource   = "data-layer"
  effect     = "allow"
  direction  = "outbound"

  conditions {
    zone.origin = "api"
    zone.target = "data-layer"
  }
}

// 没有对应的 data-layer 到 api 的规则。
// 响应流量沿已建立的连接返回。
// 来自 data-layer 到 api 的新连接被拒绝。
```

已建立连接上的响应流量仍能流动；反向的新连接被阻断。这从根本上消除了最常见的横向移动模式：低信任区域里的被攻陷服务向高信任区域发起连接。

## 错误 3：忽略了 DNS

我们做了网络分段，却没分段 DNS 解析。每个区域都能解析每一个主机名。`sandbox` 区域里被攻陷的服务连不到 `control-plane` 资源——但它能解析它们的主机名，从而获知它们的 IP 地址。

这把内部拓扑泄露给了任何获得立足点的攻击者。他们能在不发起一次跨区域连接的情况下，把整张网络地图画出来。

我们部署了按区域分发的 DNS 视图。每个区域只能解析它被授权触达的资源的主机名：

- `sandbox` 只能解析 sandbox 范围内的资源。
- `api` 能解析 data-layer、auth 与外部资源。
- `control-plane` 能解析所有资源。

DNS 如今是一条分段边界，不再只是命名服务。如果一个区域解析不到某个主机名，它就无法把那资源当作目标——连侦察都不行。

## 总结

微隔离不是拓扑练习，而是信任练习。区域应当反映信任边界、规则应当单向、配套基础设施——DNS、服务发现、证书颁发机构——必须尊重相同的边界。否则，分段只是一张满是漏洞的地图，而攻击者非常擅长找漏洞。

حين بدأنا بناء Sentinel، قيّمنا كلّ محرّك سياسات في السوق. Open Policy Agent. وCedar. وZanzibar من Google. كلّ منها صُمّم لمشكلة مختلفة، ولم يحلّ أيّ منها مشكلتنا. فبنينا Citadel.

هذه ليست قصّة "ليس مُختَرَعاً هنا". إنّها قصّة عمّا يحدث حين تتضمّن متطلّبات تقييم سياستك الزمن، وحالة الجهاز، والتاريخ السلوكي، وإعادة التقييم المستمرّ، وكلّ محرّك قائم يفترض أنّ السياسات تُقيَّم مرّة لكلّ طلب.

## مشكلة التقييم وقت الطلب

محرّكات السياسات التقليدية مُصمَّمة لدورة طلب-استجابة متزامنة. تتلقّى خدمة طلباً، وتسأل محرّك السياسات "هل ينبغي أن أسمح بهذا؟"، وتحصل على إجابة نعم أو لا. يحدث التقييم وقت الطلب، باستخدام الحقائق المتاحة في تلك اللحظة.

يعمل هذا النموذج لتفويض API. لا يعمل للتقييم المستمرّ للثقة. هذا هو السبب:

```text title="Request-time evaluation"
T=0s    User sends request → policy engine evaluates → ALLOW
T=90s   Device posture drops below threshold → nobody re-evaluates
T=180s  User sends another request → policy engine evaluates → ALLOW
        (posture score is stale — last check was 3 minutes ago)
```

أجاب محرّك السياسات بشكل صحيح في كلتا المرّتَين. المشكلة أنّ لا أحد طرح السؤال بين T=0s وT=180s. ولمدّة 180 ثانية، كان للمستخدم وصول لم يكن ينبغي أن يمتلكه.

## ما يفعله Citadel بشكل مختلف

Citadel محرّك سياسات قائم على الدفع. بدل انتظار طلب لإطلاق التقييم، يُقيّم Citadel باستمرار كلّ جلسة نشطة مقابل مجموعة السياسات الحاليّة. حين يتغيّر أيّ مُدخل، الوضع الأمني للجهاز، أو درجة السلوك، أو السياق الشبكي، أو عمر الجلسة، يُعيد Citadel التقييم فوراً ويدفع النتيجة إلى طبقة الفرض.

تعمل حلقة التقييم كلّ 90 ثانية بشكل أساسي، لكنّها تنطلق أيضاً عند الأحداث:

- يُبلّغ Watchtower عن تغيّر وضع أمني، يُعيد Citadel تقييم جميع جلسات ذلك الجهاز.
- تتغيّر حدود منطقة Rampart، يُعيد Citadel تقييم جميع الجلسات التي تعبر تلك الحدود.
- تُحدَّث سياسة ثقة، يُعيد Citadel تقييم جميع الجلسات التي تحكمها تلك السياسة.

هذا يعني أنّ الإلغاء ليس مهمّة خلفيّة. بل عمليّة آنيّة. حين يُخفق جهاز فحص وضع أمني، يُلغي Citadel الوصول خلال دورة التقييم الحاليّة، لا عند الطلب التالي.

## لغة السياسة

تُكتب سياسات Citadel بلغة خاصّة بالمجال تُسمّى Grain. صمّمنا Grain لقابليّة القراءة من قِبَل فرق الأمن، لا المطوّرين. تُقرأ سياسة Grain كإعلان نيّة:

```text title="trust-policy.grain"
policy "production-database" {
  resource = "db-cluster-prod"
  effect   = "allow"

  conditions {
    identity.role      = "engineer"
    identity.team      in ["platform", "data"]
    device.posture     >= 85
    device.encryption  = true
    session.age        <= 1800
    behavior.score     >= 75
    time.window        = "weekdays 06:00..22:00 UTC"
  }

  on_failure {
    action = "revoke"
    notify = ["security-ops", "team-lead"]
    log    = "spyglass"
  }
}
```

كلّ شرط حقيقة تُقيَّم باستمرار. `session.age <= 1800` لا يعني أنّ الجلسة كانت أقلّ من 30 دقيقة حين بدأت. بل يعني أنّ الجلسة أقلّ من 30 دقيقة الآن. حين تعبر الجلسة تلك العتبة، يُلغي Citadel الوصول دون انتظار الطلب التالي.

## ما الذي تنازلنا عنه

بناء محرّك سياسات مخصّص يعني صيانة محرّك سياسات مخصّص. لا نستطيع توجيه العملاء إلى وثائق OPA أو ساحة Cedar. نملك اللغة، والمُقيّم، ومُصحّح الأخطاء، والوثائق.

تلك المقايضة تستحقّ، لأنّ البديل، تركيب التقييم المستمرّ على محرّك يعمل وقت الطلب، كان سيتطلّب منّا بناء معظم Citadel على أيّ حال، لكن بوصفه غلافاً حول افتراضات شخص آخر. اخترنا أن نمتلك الافتراضات عوض ذلك.

النتيجة محرّك سياسات صُمّم من البداية للمشكلة التي تطرحها الثقة الصفرية فعلاً، ليست "هل ينبغي السماح بهذا الطلب؟" بل "هل ينبغي أن تظلّ هذه الجلسة موثوقة؟"

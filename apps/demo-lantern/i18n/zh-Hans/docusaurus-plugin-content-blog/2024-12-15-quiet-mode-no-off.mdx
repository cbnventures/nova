一间知道自己是卧室的卧室，不应当为"自己是一间卧室"这件事发出声响。Lantern 的 quiet 模式从来不带关闭开关——因为卧室从来不会停止做卧室。

模式会在旅行期间放松，在独自一人长周末时收紧。但规则始终如一：卧室不问问题、不发提醒、不持立场，灯只是在等。

Lantern 的第一个版本，每个分区都有一个关闭开关。卧室的开关，是我们用得最多的那一个。某个周五晚上，我们撤防卧室是为了在床上读书时不让灯每翻一页就闪一下。然后我们忘了它处于关闭状态。猫在凌晨两点穿过床面，雷阵雨期间窗户接触器开了一次，楼上安静了三小时——屋里却没人知道。我们也不知道。卧室没有意见，因为我们告诉它别有。

那个周末我们把关闭开关移除了。卧室换上了 quiet 模式——一种"灯昏暗、提示静默、仪表盘不刷新"的模式。但传感器仍在听，历史仍在写，看守逻辑仍在运行，只是没有观众而已。

配置看起来很小，因为这种理念本身就很小。

```yaml title="lantern.modes.yml"
zones:
  bedroom:
    sensors: [motion-bedroom, contact-window]
    schedule:
      always: quiet         # 从不关闭，也从不提示
    quiet-mode:
      alerts:    none       # 灯不出声
      history:   write      # 账册继续记
      handoffs:  pass       # 其他分区仍可询问
      surface:   dim        # 仪表盘的瓦块淡下来
```

Quiet 模式有四行行为。没有一行是关闭。灯昏暗，历史继续记，其他分区仍可向卧室询问看到过什么。卧室会如实回答，只是很轻。

我们从第二个周末学到了这条规则。猫又在 1:47 走过床面。窗户接触器在 3:09 报告了一次短暂开启——这次不是雷阵雨，只是猫找了一道缝隙。灯什么也没说，家里人睡过了这两件事。周日下午我们边喝咖啡边读账册，看到那一行关于窗户的记录，便上楼去看门栓。门栓需要转半圈，我们转了半圈，lantern 又安静回去了。

卧室是一个只有一条规则的房间：它是用来睡觉的。Quiet 模式就是同一条规则，只是写给灯听。关闭会更简单。安静，是这间房真正需要的。

同样的规则后来扩散到其他房间。婴儿房是第二个采用的，那是我们家庭里另一户的。第三个是早班工作的家庭成员的居家办公室。会贴上 *quiet* 标签的分区清单，刻意保持得很短——这是一份"灯可以在场但不可以发声"的房间清单。

```bash
$ lantern zones list --mode quiet
zone           since         alerts   history   handoffs   surface
bedroom        2024-12-15    none     write     pass       dim
nursery        2025-02-08    none     write     pass       off
home-office    2025-09-21    none     write     pass       dim
```

清单增长得很慢。我们已经七个月没有加入过新的 quiet 分区。决定哪些房间穿 quiet 的，是家里人；大多数房间并不需要。厨房需要被听见，过道需要在场，卧室什么都不要——灯如其所愿。这就是整件事的全部协定。

我们接触过那种把 40% 的时间花在准备审计上的安全团队——不是响应事件，不是改善安全态势，而是为只会被审计员看一眼然后归档的证据夹做准备。花在合规准备上的时间，就是没花在安全工作上的时间——这个取舍本身就是一种安全风险。

审计疲劳不是士气问题，是攻击面。

## 人工收集证据的成本

一次典型的 SOC 2 Type II 审计要求覆盖 60 到 80 个控制点的证据。对每一个控制点，安全团队都得拿出文档，证明它存在、被执行、且在整个审计周期内一直在正确运转。大多数组织里，这些证据是人工收集的：

- 访问控制配置的截图
- 身份提供方仪表盘的导出
- 把用户对应到角色再对应到资源的电子表格
- 表明监控仍在运转的日志片段
- 来自工单系统的变更管理记录
- 来自系统所有者的签字声明

每一份证据都是一个时间点上的快照。某张访问控制配置截图只表示截图当天的状态，它不告诉你昨天、上个月、或审计周期任何其他时刻里这个配置是不是同样如此。

整个人工流程，平均要让客户的工程师投入 320 个工时——也就是一名资深工程师整整 8 周的时间，而这段时间产出的安全价值为零。

## Spyglass 取代了什么

Spyglass 维护一份对 Sentinel 平台上每一件安全相关事件的持续、不可篡改的记录。它不在按需时收集证据，而是在系统正常运行的过程中持续地记录证据，作为副产品。

下面是有无 Spyglass 时的证据收集流程对比：

| 控制点     | 人工流程             | Spyglass 流程             |
|---------|------------------|-------------------------|
| 存在访问控制  | IAM 控制台截图        | 持续的策略评估日志               |
| MFA 已强制 | 身份提供方导出          | 每会话的 MFA 校验记录           |
| 日志已保留   | 存储桶清单 + 保留策略截图   | 加密链式的不可篡改日志             |
| 监控已运转   | 仪表盘截图 + 告警配置导出   | Watchtower 评估记录（90 秒节奏） |
| 变更已被复核  | Jira 工单导出 + 审批截图 | 每次变更的 Parapet 仿真记录      |
| 加密已启用   | 配置文件片段           | Rampart 隧道密钥轮换记录        |

右栏每一行都是自动生成的。安全团队不准备证据，他们执行一条查询：

```bash title="生成完整的 SOC 2 证据包"
sentinel spyglass export \
  --framework soc2-type2 \
  --period "2025-07-01..2025-12-31" \
  --format evidence-bundle
```

```text title="证据包输出"
SOC 2 Type II 证据包 —— 2025 下半年
  生成时间：2026-01-03T09:00:00Z
  时段：    2025-07-01 至 2025-12-31（184 天）

  控制类别：
    CC6: 逻辑与物理访问控制    14 项控制   PASS
    CC7: 系统运行             11 项控制   PASS
    CC8: 变更管理              8 项控制   PASS
    CC9: 风险缓解              6 项控制   PASS

  证据条目：2,847,291 条
  加密完整性：SHA-256 链已校验
  检出缺口：0（连续覆盖，无遗漏区间）

  包体大小：4.2 GB（压缩后）
  导出格式：PDF 摘要 + JSON 明细 + 原始日志归档
```

320 个工时被压缩为一条命令。证据更完整——它覆盖了审计周期的每一秒，而不是某人记得去截图的那几个瞬间。证据可加密校验，审计员可以独立确认事后没有任何记录被篡改。

## 真正的风险

安全风险不是审计很烦人，而是花在审计准备上的时间挤占了真正用来做安全的时间。我们追踪过一个客户在部署 Spyglass 前后的时间分布：

```text title="安全团队的时间分配"
Spyglass 之前：
  审计准备：       40%
  事件响应：       15%
  策略复核：       10%
  威胁建模：        5%
  工具改进：       10%
  会议与协调：     20%

Spyglass 之后：
  审计准备：        5%
  事件响应：       20%
  策略复核：       20%
  威胁建模：       15%
  工具改进：       20%
  会议与协调：     20%
```

团队没变大。同一拨人，如今把 35% 的时间花在策略复核、威胁建模与工具改进上，而不是在收集截图。这相当于把一支安全团队 35% 的精力，从合规剧场重新投到真正的防御上。

## 原则

合规证据应当是安全运营的副产品，而不是一条独立的工作流。如果你的安全做对了——把决策记录下来、持续评估策略、按节奏轮换密钥、变更前先仿真——审计证据就已经存在了。你需要的，只是一个会持续记录、按需导出的系统。

这就是 Spyglass 在做的事。不是因为审计本身重要（它确实重要），而是因为反过来——一个把 40% 时间花在收集证据上的安全团队，是一个在真正职责上效率打了六折的安全团队。

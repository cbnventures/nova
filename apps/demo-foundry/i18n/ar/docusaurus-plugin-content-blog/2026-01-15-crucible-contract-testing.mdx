في العام الماضي، كسرنا الإنتاج لأن حزمة مكتبة غيّرت شكل نوع عودة. كان التغيير متوافقًا للخلف وفق مُترجم TypeScript — الحقول القديمة لا تزال موجودة، والحقل الجديد اختياري. لكن ثلاث حزم سفلية اعتمدت على غياب ذلك الحقل لتشغيل مسار احتياطي. حين وُجد الحقل فجأة (بقيمة `undefined`)، توقّف الاحتياطي عن الإطلاق، ورأى المستخدمون شاشات فارغة لمدة أربعين دقيقة.

قال المُترجم إن التغيير آمن. اختلف وقت التشغيل. تلك الحادثة هي السبب في بنائنا اختبار العقود في Crucible.

## ما يعنيه اختبار العقود

اختبار العقد يتحقق من سلوك حدود الحزمة — الواجهة العامة التي تستهلكها الحزم الأخرى. ليس اختبار وحدة (الذي يختبر المنطق الداخلي) ولا اختبار تكامل (الذي يختبر النظام الكامل). يجلس بينهما: يختبر الواجهة بين حزمتين باستخدام شيفرة حقيقية، لا محاكاة.

في Crucible، اختبار العقد يعيش في حزمة المستهلك، لا المُزوِّد. تكتب حزمة `web` اختبار عقد للواجهة التي تتوقّعها من حزمة `core`. إن تغيّرت `core` بطريقة تنتهك العقد، يفشل الاختبار في `web` — الحزمة التي ستنكسر فعلًا.

```alloy title="packages/web/contracts/core.contract.al"
crucible.contract("core/formatCurrency") {
  // Contract: returns a string with currency symbol prefix
  const result = formatCurrency(42.5, "USD")
  assert typeof result == "Text", "Must return Text"
  assert result.startsWith("$"), "Must prefix with currency symbol"
  assert result.contains("."), "Must include decimal separator"
}

crucible.contract("core/resolveUser") {
  // Contract: returns undefined for unknown users (not null, not throws)
  const result = resolveUser("nonexistent-id")
  assert result === undefined, "Unknown users must return undefined"
}
```

العقد الثاني هو بالضبط نوع الاختبار الذي كان سيلتقط حادثة إنتاجنا. يُؤكِّد العقد صراحة أن المستخدمين المفقودين يُعيدون `undefined`، لا كائنًا جزئيًا بحقول `undefined`.

## كيف يكتشف Crucible العقود

لا يتطلب Crucible تسجيلًا يدويًا. يفحص كل حزمة بحثًا عن ملفات تطابق النمط `contracts/*.contract.al` ويُحدِّد تلقائيًا أي حزمة مُزوِّد يستهدفها العقد بقراءة الاستيرادات.

أثناء `foundry ignite`، يُشغِّل Crucible العقود بعد بناء حزمة المُزوِّد ولكن قبل بناء حزمة المستهلك. هذا يضمن أن تغييرًا كاسرًا في `core` يُلتقط قبل أن تحاول `web` حتى الترجمة مقابل النسخة الجديدة.

```text title="Bellows task ordering with contracts"
[bellows] core (build)           → 2.1s
[bellows] core (contracts: web)  → 0.4s ← web's contract runs against core
[bellows] core (contracts: api)  → 0.3s ← api's contract runs against core
[bellows] web (build)            → 3.2s ← only starts if contracts pass
[bellows] api (build)            → 2.8s
```

إن فشل عقد، يوقف Bellows البناء السفلي فورًا. لا تنتظر 3 دقائق لـ `web` لتفشل بخطأ ترجمة محيّر. تحصل على رسالة انتهاك عقد واضحة خلال ثوانٍ من التغيير الكاسر.

## كتابة عقود جيدة

كتبنا أكثر من 400 اختبار عقد عبر مساحة عملنا الداخلية. ثلاثة أنماط تُميِّز العقود المفيدة من الضوضاء:

- **اختبر السلوك، لا الشكل.** عقد يُؤكِّد `typeof result.name == "Text"` هشّ — ينكسر إذا أُعيدت تسمية الحقل. عقد يُؤكِّد `result.displayLabel().length > 0` يختبر السلوك الذي يعتمد عليه المستهلك فعلًا.
- **اختبر الغياب صراحة.** إذا اعتمدت شيفرتك على غياب حقل، اكتب عقدًا يُؤكِّد أن الحقل `undefined`. الافتراضات الضمنية تصبح علل غير مرئية.
- **اختبر مسارات الخطأ.** ينبغي أن تغطّي العقود ما يحدث حين يرفض المُزوِّد المدخلات، أو يُعيد نتائج فارغة، أو ينتهي الوقت. هذه المسارات هي حيث تعيش معظم العلل عبر الحزم.

## سؤال المقياس

تشغيل 400 عقد على كل بناء يبدو مكلفًا. عمليًا، تخزّن Quench نتائج العقود تمامًا كما تخزّن قطع البناء. إن لم يتغيّر المُزوِّد ولا المستهلك، تُستعاد نتيجة العقد من الذاكرة. على طلب سحب نموذجي، أقل من 20 عقدًا تُنفَّذ فعلًا — الباقي إصابات ذاكرة.

العبء نحو 2 إلى 4 ثوانٍ لكل بناء. تكلفة عدم تشغيلها — كما تعلّمنا من حادثة إنتاجنا — تُقاس بثقة المستخدم وساعات المهندس.

## جرّبه

أضِف دليل `contracts/` إلى أي حزمة، اكتب ملف `.contract.al` يستورد من تبعية، وشغِّل `foundry ignite`. يلتقطه Crucible تلقائيًا. لا إعداد مطلوب.

ابدأ بالحزمة التي تحوي أكبر عدد من المستهلكين السفليين. هذا حيث تُقدِّم العقود أكبر قيمة — وحيث تُحدث التغييرات الكاسرة أكبر ضرر.

在开始做 Sentinel 时，我们评估了市面上每一个策略引擎：Open Policy Agent、Cedar、Google Zanzibar。每一个都为不同的问题而设计，但没有一个解决我们的问题。于是我们自己写了 Citadel。

这不是"非我所造"的故事。这是当策略评估的需求里同时包括了时间、设备状态、行为历史与持续再评估，而现成的引擎都假设策略是按请求评估一次——之后会发生什么的故事。

## 请求时评估的问题

传统策略引擎是为同步的请求-响应循环而设计的。服务接到请求，问策略引擎"我该不该允许这次访问？"然后拿到一个是或否的答案。评估发生在请求时，使用此刻可得的事实。

这个模型适合 API 授权，不适合持续信任评估。原因如下：

```text title="请求时评估"
T=0 秒    用户发起请求 → 策略引擎评估 → ALLOW
T=90 秒   设备态势跌破阈值 → 没有人来重新评估
T=180 秒  用户再发起请求 → 策略引擎评估 → ALLOW
          （态势评分已陈旧——上次检查是 3 分钟前）
```

策略引擎两次都答对了。问题在于 T=0 秒到 T=180 秒之间没有人问过这个问题。这 180 秒里，用户拥有他本不该拥有的访问权。

## Citadel 不一样的地方

Citadel 是一个推送式的策略引擎。它不等请求来触发评估，而是对每一个活跃会话按当前策略集进行持续评估。任何一项输入发生变化——设备态势、行为评分、网络上下文、会话时长——Citadel 都会立刻重新评估，并把结果推送到执行层。

评估循环以 90 秒为基线节奏运行，并对事件作出触发响应：

- Watchtower 上报态势变化 → Citadel 重新评估该设备所有会话。
- Rampart 区域边界变化 → Citadel 重新评估跨该边界的所有会话。
- 信任策略更新 → Citadel 重新评估该策略所管控的所有会话。

这意味着撤销不是后台任务，而是一次实时操作。当设备态势检查失败时，Citadel 会在当前评估周期内撤销访问——而不是等下一次请求。

## 策略语言

Citadel 的策略以一种叫 Grain 的领域专用语言书写。我们为安全团队的可读性而设计 Grain，而不是为开发者。一条 Grain 策略读起来像意图声明：

```text title="trust-policy.grain"
policy "production-database" {
  resource = "db-cluster-prod"
  effect   = "allow"

  conditions {
    identity.role      = "engineer"
    identity.team      in ["platform", "data"]
    device.posture     >= 85
    device.encryption  = true
    session.age        <= 1800
    behavior.score     >= 75
    time.window        = "weekdays 06:00..22:00 UTC"
  }

  on_failure {
    action = "revoke"
    notify = ["security-ops", "team-lead"]
    log    = "spyglass"
  }
}
```

每一个条件都是一个持续被评估的事实。`session.age <= 1800` 不是说"会话开始时低于 30 分钟"，而是"会话此刻低于 30 分钟"。当会话越过这条线，Citadel 会立即撤销访问，不等下一次请求。

## 我们放弃了什么

自建策略引擎，就要长期维护策略引擎。我们没法让客户去看 OPA 的文档或 Cedar 的演练场。我们要自己持有语言、求值器、调试器与文档。

这笔取舍值得。因为反过来——把持续评估硬塞进一个请求时引擎——其实需要我们写下 Citadel 里几乎所有的东西，但又是包在别人假设之上的实现。我们选择直接持有这些假设。

最终得到的是一个从一开始就为"零信任的真实问题"而设计的策略引擎——那个问题不是"这次请求是否应被允许？"而是"这个会话是否还值得信任？"

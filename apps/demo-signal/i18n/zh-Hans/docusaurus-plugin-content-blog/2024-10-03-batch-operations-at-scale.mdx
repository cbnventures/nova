一家金融科技客户带着一个问题来找我们。他们在三家不同的短链服务上有 140,000 条活跃短链。每一家都有自己的重定向规则、自己的分析孤岛、自己的品牌域名。客户希望把所有东西整合进 Beacon——并且不能弄断任何一条现存的链接。

Relay 在十一分钟里处理完了整次迁移。零宕机。零失效重定向。完整保留了历史归因数据。

## 迁移流水线

Relay 是 Signal 的批量操作引擎。它通过一份声明式作业格式来处理批量链接创建、修改与迁移：

```json title="Relay 迁移作业"
{
  "job": "migrate",
  "source": {
    "provider": "bitly",
    "apiKey": "env:BITLY_API_KEY",
    "filter": {
      "createdAfter": "2022-01-01",
      "status": "active"
    }
  },
  "target": {
    "domain": "go.fintech.example",
    "preserveSlugs": true,
    "attribution": {
      "campaign": "migration-q4",
      "channel": "legacy"
    }
  },
  "options": {
    "dryRun": false,
    "concurrency": 50,
    "redirectChain": true
  }
}
```

`redirectChain` 选项至关重要。它会把旧短链配置成先经过 Beacon 再到达最终目的地。点击老链接的用户被无缝路由进新系统，从那一刻起，这些点击就会进入 Beacon 的分析数据。

## 大规模下的性能

我们在不同作业规模下做了 Relay 的基准测试：

| 链接数     | 用时      | 吞吐量      | 错误率   |
|---------|---------|----------|-------|
| 1,000   | 4s      | 250 链接/秒 | 0.00% |
| 10,000  | 38s     | 263 链接/秒 | 0.00% |
| 50,000  | 3m 12s  | 260 链接/秒 | 0.01% |
| 140,000 | 11m 04s | 211 链接/秒 | 0.02% |

吞吐量在大约 50,000 条之前保持稳定。再往上，瓶颈变成了源服务商 API 的限速——而不是 Relay 自身。140,000 条规模下的错误率是 0.02%，也就是 28 条链接因源数据中的目的地 URL 拼写有问题而需要人工复核。

## 批量修改

迁移只是一种用法。更常见的 Relay 操作是批量修改——当一次营销活动调整、或品牌域名轮换时，需要一次性更新成千上万条链接：

- **域名轮换**：把所有链接从 `go.old-brand.com` 迁到 `go.new-brand.com`，一次作业搞定。
- **归因回填**：为历史上没带归因元数据的旧链接补上营销活动信息。
- **过期清扫**：为超过某个时长的旧链接统一设置过期时间。
- **目的地更新**：把所有指向已弃用 URL 的链接重定向到它的替代地址。

每一项操作都作为一个原子作业运行，并完整支持回滚。如果作业中途失败，Relay 会撤销部分更改，并精确报告哪些链接未能处理。

## 为什么"批量"很重要

大多数链接管理工具把链接当作单个对象来对待。你一条条创建、一条条编辑、一条条分析。这套打法在你只有一百条链接时还能用，但在十万条链接时就站不住了。

Relay 把链接视为一份数据集。你以声明的方式描述操作，Relay 对照完整链接清单做校验，然后执行。"一条条管理链接"与"以批量方式管理链接"之间的区别，就是"在表格里一格格改"与"跑一次 SQL 迁移"之间的区别。
